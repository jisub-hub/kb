# 개발 산출물 (Software Deliverables / SDLC Artifacts)

> [!summary] 한 줄 요약
> **산출물(Deliverable)** 은 SDLC 각 단계가 만들어내는 결과물(문서·코드·테스트·매뉴얼)로, **소통·추적성·검수·유지보수**의 근거가 된다. 표준은 ISO/IEC 12207(소프트웨어 생명주기 프로세스)과 IEEE 830(SRS)이 제시한다. 핵심은 "얼마나 많이"가 아니라 **추적성을 확보하되 문서를 위한 문서를 피하는 균형**이다.

---

## 1. 개요

**산출물**이란 SDLC(Software Development Life Cycle)의 각 단계에서 생산되는 검증 가능한 결과물이다. 코드만이 아니라 그 코드를 둘러싼 문서·테스트·매뉴얼 전부를 포함한다.

**왜 필요한가**

| 목적 | 설명 |
|------|------|
| **소통** | 기획·개발·QA·고객 간 같은 그림 공유 |
| **추적성(traceability)** | 요구 → 설계 → 코드 → 테스트가 서로 연결되는지 |
| **검수(acceptance)** | 발주처/감리가 "약속한 것을 만들었는가" 확인 |
| **유지보수** | 인수인계·장애 대응 시 "왜 이렇게 만들었나" 복원 |

**표준 근거**

- **ISO/IEC 12207** — 소프트웨어 생명주기 프로세스 국제 표준. 어떤 단계에서 어떤 산출물을 내는지의 골격.
- **IEEE 830** — SRS(Software Requirements Specification) 작성 표준. 요구사항 문서의 사실상 기준(후속 ISO/IEC/IEEE 29148로 통합).

> 산출물은 **목적**이 아니라 **수단**이다. "이 문서가 누구의 어떤 결정을 돕는가"를 답하지 못하면 만들 이유가 없다.

---

## 2. SDLC 단계별 산출물

| 단계 | 대표 산출물 | 주 작성자 | 핵심 질문 |
|------|------------|----------|----------|
| **기획** | 사업계획서, 제안요청서(RFP), 제안서, WBS, 일정계획 | PM/기획 | 무엇을 왜 만드나 |
| **요구분석** | 요구사항정의서(SRS), 유스케이스 명세, 요구사항 추적 매트릭스(RTM) | BA/기획 | 무엇이 필요한가 |
| **설계** | 아키텍처 설계서, 상세설계서, ERD, API 명세, 화면설계서, UML 다이어그램 | 아키텍트/개발 | 어떻게 만드나 |
| **구현** | 소스코드, 빌드 스크립트, 단위 테스트 코드, 코드 리뷰 기록 | 개발 | 실제 코드 |
| **테스트** | 테스트 계획서, 테스트 케이스, 테스트 결과서, 결함 리포트, 커버리지 | QA | 제대로 동작하나 |
| **배포** | 배포 가이드, 릴리스 노트, 환경 구성도, 롤백 절차 | DevOps | 어떻게 올리나 |
| **운영** | 운영자 매뉴얼, 사용자 매뉴얼, 형상관리 대장, 장애 대응 가이드(런북) | 운영 | 어떻게 유지하나 |

> 폭포수에서는 단계별로 산출물이 "완비"되어야 다음으로 넘어가지만, 애자일에서는 스프린트마다 일부만 점진적으로 만든다(7절).

## 3. 요구사항 단계

### 3.1 요구사항정의서 (SRS, IEEE 830 구조)

```
1. 소개          — 목적, 범위, 용어 정의, 참고 문헌
2. 전체 설명      — 제품 관점, 사용자 특성, 제약, 가정
3. 상세 요구사항
   3.1 기능 요구(Functional)     — "시스템은 ~해야 한다"
   3.2 비기능 요구(Non-functional) — 성능, 보안, 가용성, 확장성, 사용성
   3.3 인터페이스 요구            — UI, 외부 시스템 연동
4. 부록
```

- **기능 요구**: 무엇을 하는가(주문 생성, 결제, 알림 발송…).
- **비기능 요구**: 얼마나 잘 하는가(응답 1초 이내, 99.9% 가용성). **간과되기 쉬우나 아키텍처를 좌우**한다.

### 3.2 유스케이스 명세

액터·사전조건·기본 흐름·대안 흐름·사후조건으로 시나리오를 서술. → [[UML-Modeling]]의 유스케이스 다이어그램과 짝.

### 3.3 요구사항 추적 매트릭스 (RTM)

각 요구가 **설계·코드·테스트**로 이어지는지를 행렬로 추적. 누락·과잉 구현을 방지하는 추적성의 핵심 도구.

| 요구 ID | 요구 내용 | 설계 문서 | 구현 모듈 | 테스트 케이스 |
|---------|----------|----------|----------|--------------|
| FR-001 | 회원 가입 | DD-Auth-3 | `SignupService` | TC-101, TC-102 |
| NFR-01 | 로그인 1초 이내 | AD-2 | `AuthController` | TC-Perf-01 |

## 4. 설계 단계

| 산출물 | 내용 | 비고 |
|--------|------|------|
| **아키텍처 설계서** | 시스템 구성, 계층, 기술 스택, 배포 토폴로지 | 비기능 요구 충족 근거 |
| **상세설계서** | 모듈/클래스 단위 로직, 데이터 흐름 | 폭포수에서 두꺼움 |
| **ERD (논리/물리)** | 엔티티-관계. 논리=업무 개념, 물리=실제 테이블/컬럼/인덱스 | DB 설계 |
| **API 명세** | 엔드포인트·요청/응답·에러 코드 | OpenAPI(Swagger)로 코드와 동기화 권장 |
| **화면설계서** | 와이어프레임·스토리보드·화면 흐름 | 기획-디자인-개발 합의 |
| **UML 다이어그램** | 클래스/시퀀스/상태 다이어그램 | → [[UML-Modeling]] |

```yaml
# API 명세는 별도 문서보다 OpenAPI로 — 코드/테스트와 동기화
paths:
  /orders:
    post:
      summary: 주문 생성
      responses:
        '201': { description: 생성됨 }
        '400': { description: 잘못된 요청 }
```

> 설계 문서는 **코드와 멀어지는 순간 거짓말**이 된다. ERD는 마이그레이션에서, API는 OpenAPI에서 생성하는 식으로 "코드가 진실원천"이 되게 하면 동기화 비용이 준다(9절).

## 5. 구현 / 테스트

**구현 산출물**

- 소스코드(VCS 관리), 빌드/배포 스크립트, 의존성 명세, 코드 리뷰 기록.
- 코드 자체가 가장 정확한 산출물 — 주석·커밋 메시지·PR 설명이 "왜"를 보존.

**테스트 산출물**

| 산출물 | 내용 |
|--------|------|
| **테스트 계획서** | 범위·전략·환경·일정·합격 기준 |
| **테스트 케이스** | 입력·기대결과·실제결과(단위/통합/E2E) |
| **테스트 결과서** | 수행 결과, Pass/Fail, 결함 목록 |
| **커버리지 리포트** | 코드 중 테스트가 닿은 비율 |

```
단위 테스트   — 함수/클래스 단위, 빠르고 많이
통합 테스트   — 모듈 간 연동, DB·외부 API 포함
커버리지      — 라인/브랜치 80%대가 흔한 목표 (숫자 자체가 목적은 아님)
```

> 커버리지는 **품질의 하한 신호**일 뿐 상한 보장이 아니다. 100%를 위한 의미 없는 테스트는 "문서를 위한 문서"의 테스트 버전.

## 6. 배포 / 운영

| 산출물 | 목적 | 핵심 내용 |
|--------|------|----------|
| **배포 가이드** | 재현 가능한 배포 | 환경 구성, 배포 순서, 롤백 절차 |
| **릴리스 노트** | 변경 이력 공유 | 신규 기능, 버그 수정, 호환성 |
| **운영자 매뉴얼** | 운영팀 인수인계 | 모니터링, 백업, 장애 대응(런북) |
| **사용자 매뉴얼** | 최종 사용자 | 화면별 사용법, FAQ |
| **형상관리 대장** | 산출물 버전 추적 | 문서/코드 버전, 변경 이력, 승인자 |

> 배포 가이드의 진짜 시험대는 "**작성자 없이도 새 인원이 그대로 따라 배포할 수 있는가**". IaC(Terraform 등)·CI/CD 파이프라인이 가이드 문서를 실행 가능한 산출물로 대체하는 추세.

## 7. 방법론별 산출물 차이

| 구분 | 폭포수(Waterfall) | 애자일(Agile) |
|------|-------------------|----------------|
| 철학 | 단계별 문서 완비 후 진행 | **동작하는 소프트웨어 > 포괄적 문서** |
| 요구 | 두꺼운 SRS, 초기 확정 | User Story, Product Backlog |
| 진척 | 단계 게이트 통과 | 번다운 차트, 스프린트 데모 |
| 산출물 무게 | 무겁고 사전 완비 | 경량·점진적, "꼭 필요한 만큼(JIT)" |
| 변경 | 비싸고 어려움 | 매 스프린트 수용 |

```
애자일의 경량 산출물
- User Story    : "사용자로서 ~하고 싶다, 그래야 ~할 수 있다"
- Backlog       : 우선순위 매긴 할 일 목록
- Burndown      : 남은 작업량의 시간별 감소 추이
- DoD/AC        : 완료 정의 / 인수 기준
```

> 애자일도 "문서 없음"이 아니다. **유지보수에 꼭 필요한 최소 문서**(아키텍처 결정 기록 ADR, API 명세, 운영 런북)는 남긴다. → [[Development-Methodologies]]

## 8. 한국 SI / 공공 정보화사업 맥락

공공·SI 사업은 **감리(audit)** 와 **검수**를 전제로 산출물 표준이 강하게 규정된다(과거 정보화사업 산출물 가이드라인 등).

```
공공/SI 산출물의 현실
✅ 추적성 확보       — RTM, 요구↔테스트 매핑은 검수·분쟁 방지에 실제로 유효
✅ 인수인계 안전망    — 외주·턴키 종료 후 운영 이관 시 문서가 유일한 단서
⚠️ 과한 문서화의 함정 — 코드와 따로 노는 수백 장 문서, "감리용"으로만 쓰고 버려짐
⚠️ 형식주의          — 템플릿 채우기가 목적이 되어 정작 설계 사고가 빠짐
```

**실무 균형 조언**

- 감리·검수에 **반드시 요구되는 문서**와 **팀이 실제로 보는 문서**를 구분하라.
- 가능하면 **단일 원천에서 생성**(OpenAPI→API 문서, 마이그레이션→ERD)해 이중 관리를 줄여라.
- "이 문서를 6개월 뒤 누군가 읽는가"로 보존 가치를 판단하라.

## 9. 산출물 관리

| 이슈 | 문제 | 대응 |
|------|------|------|
| **버전관리** | 어느 문서가 최신인지 모름 | VCS·위키로 문서도 버전 추적, 형상관리 대장 |
| **SSOT (단일 진실원천)** | 같은 정보가 여러 문서에 중복 → 불일치 | 정보마다 진실원천 하나만, 나머지는 참조/링크 |
| **코드-문서 동기화** | 코드는 바뀌는데 문서는 안 바뀜 → 거짓 문서 | 코드에서 문서 생성, 변경 시 문서 갱신을 DoD에 포함 |

```
동기화 비용을 줄이는 방향
- 변하는 사실(API 스펙, 스키마)  → 코드/스키마에서 자동 생성
- 변하지 않는 의도(왜 이 설계)   → ADR·문서로 명시 보존
```

> 가장 흔한 실패는 **문서와 코드의 분기(divergence)**. "최신이 아닌 문서는 없느니만 못하다." 사실은 코드/스키마에서 뽑고, 사람이 쓰는 문서는 **변하지 않는 의도**에 집중한다.
