# 개발 방법론 (Software Development Methodologies)

> [!summary] 한 줄 요약
> 소프트웨어를 **어떤 프로세스로 만들 것인가**에 대한 모델. 단일 패스 계획 주도(폭포수)에서 위험 주도(스파이럴)를 거쳐 적응적 반복(애자일)으로 진화해왔다. "무엇이 옳다"가 아니라 **프로젝트 불확실성·규모·규제에 따라 고른다.**

---

## 1. 방법론의 진화 — 큰 그림

```
1970  Waterfall (Royce)        — 단계적·계획 주도, 문서 중심
  │   (Royce는 사실 단일 패스의 위험을 경고하고 "반복"을 권했다 — 아래 역설)
1980s Iterative/Incremental    — 단계를 반복·점증 (Larman & Basili 정리)
1988  Spiral (Boehm)           — 위험 주도, 반복마다 리스크 평가
2001  Agile Manifesto          — 변화 수용, 동작하는 소프트웨어, 짧은 주기
2010s DevOps/지속적 전달        — 개발-운영 통합, 자동화 파이프라인
```

축은 두 개다: **계획 주도(plan-driven) ↔ 적응적(adaptive)**, **문서 중심 ↔ 동작 소프트웨어 중심**.

---

## 2. Waterfall (폭포수) — Royce 1970

```
요구분석 → 설계 → 구현 → 검증(테스트) → 유지보수
   각 단계 완료 후 다음으로 (위에서 아래로 흐름, 역류 없음)
```

- **특징**: 단계별 산출물 명확, 진척 가시성, 검수 용이.
- **한계**: 요구가 초기에 확정돼야 함 → 후반 변경 비용 폭증, 동작하는 결과가 늦게 나옴.
- **적합**: 요구가 안정적·규제가 강한 도메인(국방·금융 일부·공공 SI).

> [!important] Royce의 역설
> 흔히 "폭포수의 창시자"로 인용되는 Royce(1970)는 **오히려 단일 패스 순차 방식의 위험을 경고**했다. 그는 "적어도 두 번은 반복하라(do it twice)"며 프로토타이핑·피드백을 권했다. 즉 우리가 아는 순수 폭포수는 그의 논문이 *비판한* 모델에 가깝다 — 후대가 앞부분 다이어그램만 떼어 표준처럼 굳혔다. [Royce, 1970]

### V-Model (폭포수 변형)
검증(verification)과 확인(validation)을 강조 — 각 개발 단계에 대응하는 테스트 단계를 좌우 대칭으로 배치(요구↔인수테스트, 설계↔통합테스트, 상세설계↔단위테스트).

---

## 3. Iterative & Incremental (반복·점증)

폭포수 단계를 **여러 번 반복**하며 시스템을 조금씩(increment) 완성. 사실 1950~70년대부터 실무에 존재했고 Larman & Basili(2003)가 그 역사를 정리했다.

```
반복(Iteration): 각 주기에 분석→설계→구현→테스트를 작게 수행
점증(Increment): 주기마다 동작하는 기능 조각을 추가
→ 위험을 일찍 발견, 피드백 반영. 애자일·스파이럴의 토대.
```

---

## 4. Spiral (스파이럴) — Boehm 1988

**위험 주도(risk-driven) 프로세스 모델 생성기.** 나선을 돌 때마다 4개 활동을 반복하며, 가장 위험이 큰 부분을 먼저 해결한다.

```
4사분면 (나선 1회전):
  ① 목표·대안·제약 식별
  ② 대안 평가, 위험 식별·분석 (← 핵심: 프로토타입으로 위험 완화)
  ③ 개발·검증
  ④ 다음 단계 계획·검토

→ 위험이 클수록 더 많은 나선(분석·프로토타이핑)을 돈다
→ "반복되는 폭포수 + 매 반복마다 리스크 평가"로 이해 가능
```

- **적합**: 대규모·고위험·요구 불확실 프로젝트.
- **한계**: 위험 분석 전문성 요구, 작은 프로젝트엔 과함. [Boehm, 1988]

## 5. Agile (애자일) — Manifesto 2001

2001년 17인이 **애자일 선언(Manifesto)**을 발표. 4가지 가치:

```
프로세스·도구       < 개인과 상호작용
포괄적 문서         < 동작하는 소프트웨어
계약 협상           < 고객과의 협력
계획 준수           < 변화에 대한 대응
(왼쪽도 가치 있지만, 오른쪽을 "더" 중시)
```

12원칙의 핵심: 짧은 주기 잦은 인도, 변화 환영, 자기조직 팀, 지속 가능한 속도, 동작하는 소프트웨어가 진척의 척도.

### 5.1 Scrum
```
스프린트(1~4주) 반복. 역할(PO/SM/개발팀),
산출물(제품 백로그/스프린트 백로그/증분),
이벤트(스프린트 계획/데일리/리뷰/회고)
```

### 5.2 XP (eXtreme Programming)
기술 실천 강조 — TDD, 페어 프로그래밍, 지속적 통합(CI), 리팩터링, 단순 설계. [Beck, 1999]

### 5.3 Kanban
WIP(진행 중 작업) 제한 + 흐름 시각화(보드). 반복(스프린트)보다 **연속적 흐름**. 운영·유지보수에 적합.

## 6. 비교표

| 방법론 | 주도 | 변화 대응 | 산출물 | 위험 처리 | 적합 |
|--------|------|----------|--------|----------|------|
| Waterfall | 계획 | 약함 | 무거움 | 늦음 | 안정 요구·규제 |
| V-Model | 계획 | 약함 | 무거움 | 테스트 강화 | 임베디드·검증 중요 |
| Iterative | 혼합 | 중간 | 중간 | 점진 | 중대형 |
| Spiral | 위험 | 중간 | 중간 | **선제적** | 대규모·고위험 |
| Agile/Scrum | 적응 | **강함** | 경량 | 매 스프린트 | 불확실·변화 잦음 |
| Kanban | 흐름 | 강함 | 경량 | 연속 | 운영·지원 |

## 7. 선택 기준 — "균형"

Boehm & Turner(2003)는 *애자일 vs 규율(plan-driven)*이 이분법이 아니라 **스펙트럼**이며, 프로젝트 특성으로 균형점을 찾으라 했다.

```
애자일 쪽이 유리: 요구 불확실·변화 잦음, 작은 팀, 빠른 피드백 가능, 낮은 규제
계획 주도 유리:   요구 안정, 대규모·분산 팀, 높은 신뢰성/규제(인증·감사), 고비용 실패
현실:            대부분 하이브리드 (애자일 실행 + 필수 문서·게이트 유지)
```

> 방법론은 신념이 아니라 도구다. "왜 이 방법이 이 프로젝트에 맞는가"를 설명할 수 있어야 한다.
