# 팀 커뮤니케이션 & 보고 체계

> [!summary] 한 줄 요약
> 개발팀의 커뮤니케이션은 **비동기(문서) + 동기(미팅) 혼합**이 효율적이다. 보고는 계층보다 컨텍스트 공유가 목적이다.

---

## 1. 보고 체계 (Reporting Structure)

### 일일 스탠드업 (Daily Standup)

```
형식: 15분 이내, 슬랙 텍스트 또는 화상
시간: 오전 업무 시작 후 30분 이내

템플릿:
  어제: [완료한 것]
  오늘: [할 것]
  블로커: [막힌 것 / 도움 필요 여부]

원칙:
  - "진행 중"이 아니라 "완료 / 시작 / 막힘"으로 명확히
  - 상세 논의는 스탠드업 후 별도로
  - 원격 팀은 매일 오전 슬랙에 비동기 포스팅도 OK
```

### 주간 보고

```
형식: 팀 위키 또는 노션 페이지 업데이트
작성자: 개발 리드 → PM / 팀장

포함 내용:
  - 이번 주 완료된 기능/수정
  - 다음 주 계획 (스프린트 기준)
  - 이슈 및 리스크 (빨간불 항목)
  - 지표 (배포 횟수, 버그 발생수, SLA 준수율)
```

### 에스컬레이션 (문제 보고)

```
에스컬레이션 타이밍:
  - 혼자 2시간 이상 막혔을 때 → 팀원 도움 요청
  - 스프린트 목표 영향 가능성 → 리드/PM 즉시 공유
  - 프로덕션 장애 → 즉시 [#incident 채널] + 리드 멘션

에스컬레이션 형식:
  상황: 어떤 문제인가
  영향: 어떤 기능/사용자에게 영향
  현재 시도한 것: 무엇을 했는지
  필요한 것: 도움/결정/자원
```

## 2. 미팅 문화

### 미팅 유형별 운영

| 미팅 | 주기 | 시간 | 목적 |
|------|------|------|------|
| 데일리 스탠드업 | 매일 | 15분 | 진행 상황 공유, 블로커 확인 |
| 스프린트 플래닝 | 2주 1회 | 2시간 | 다음 스프린트 작업 선정 |
| 스프린트 리뷰 | 2주 1회 | 1시간 | 완료된 기능 데모 |
| 회고 (Retro) | 2주 1회 | 1시간 | 팀 프로세스 개선 |
| 설계 논의 | 필요 시 | 1시간 | 큰 기능 설계 합의 |
| 1:1 | 매주 | 30분 | 개인 성장, 피드백 |

### 미팅 운영 원칙

```
미팅 전:
  □ 아젠다 공유 (최소 하루 전)
  □ 사전 자료 있으면 미팅 전에 읽어오기
  □ 불필요 참석자 제외 (결정권자 + 관련자만)

미팅 중:
  □ 서기(노트 담당) 지정
  □ 결정 사항은 실시간으로 문서화
  □ 논쟁이 길어지면 "오프라인으로" 또는 "별도 미팅"

미팅 후:
  □ 24시간 이내 회의록 슬랙 공유
  □ 액션 아이템: 담당자 + 마감일 명시
  □ 결정 안 난 건: 다음 미팅 전 결론 목표 설정
```

## 3. 채널 / 슬랙 운영 규칙

```
#general         팀 전체 공지, 중요 결정 공유
#dev             개발 관련 논의, 기술 질문
#deploy          배포 알림, 배포 실패 알림 (Bot 자동화)
#incident        프로덕션 장애 (즉각 알림)
#random          자유 대화

원칙:
  - 업무 DM 지양 → 채널 대화로 (히스토리 공유)
  - 스레드(Thread) 활용 — 채널 노이즈 줄이기
  - @here / @channel 은 긴급 상황만
  - 응답 기대 시간: 슬랙 2시간 이내, 이메일 24시간 이내
```

## 4. 문서화 규칙

### 무엇을 문서화하는가

```
✅ 반드시 문서화:
  - 아키텍처 결정 (ADR — Architecture Decision Record)
  - API 명세 (Swagger, Notion)
  - 온보딩 가이드 (로컬 실행, 배포 방법)
  - 장애 사후 분석 (Post-mortem)
  - 인프라 구성 (다이어그램 + 설명)

❌ 문서화 불필요:
  - 코드가 자명한 로직 (코드 = 문서)
  - 일시적 메모 (슬랙 스레드로 충분)
  - 이미 Git 히스토리에 있는 것
```

### ADR (Architecture Decision Record)

```markdown
# ADR-001: 인증 방식으로 JWT Stateless 선택

## 상태
채택됨 (2026-01-15)

## 컨텍스트
서비스가 다중 인스턴스로 배포됨. 세션 공유를 위한 Redis 추가는 운영 복잡도 증가.

## 결정
JWT Stateless 인증 채택. 토큰 만료는 15분, Refresh Token은 Redis에 저장.

## 결과
장점: 수평 확장 용이, 인스턴스 간 상태 공유 불필요
단점: 토큰 즉시 무효화 불가 (로그아웃 = 클라이언트 삭제만)
      → Refresh Token Redis 블랙리스트로 부분 해결
```

## 5. 온보딩 체크리스트

```
신규 팀원 첫 주:
  □ 코드 컨벤션 문서 읽기
  □ 로컬 개발 환경 설정 (README 기준)
  □ 슬랙 채널 가입
  □ Github 저장소 접근권한 확인
  □ 배포 환경(dev/stg/prod) 접근 확인
  □ 스프린트 플래닝 참석
  □ 기존 코드 PR 리뷰 관찰 (코드 읽기)

첫 PR 기준:
  □ 작은 범위 (버그 수정 또는 문서 개선)
  □ 코딩 컨벤션 숙지 후 작성
  □ 리드의 pair review 진행
```

## 6. 갈등 해결 원칙

```
기술 논쟁:
  1. 데이터와 근거로 논의 (감정 제외)
  2. 합의 안 되면 → 프로토타입/PoC로 검증
  3. 최종 결정자 지정 (팀 리드 또는 아키텍트)
  4. 결정 후에는 따름 (개인 의견은 ADR에 기록)

커뮤니케이션 충돌:
  1. 채널에서 논쟁 → 1:1로 옮기기
  2. 1:1 해결 안 되면 → 팀 리드에게 중재 요청
  3. 공개 비난/인신공격은 즉시 중단
```
