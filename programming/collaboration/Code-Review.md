# 코드 리뷰 (Code Review)

> [!summary] 한 줄 요약
> 코드 리뷰는 버그 찾기보다 **지식 공유와 설계 합의**에 가치가 있다. 코드를 비판하되 사람을 비판하지 않는다.

---

## 1. 리뷰어의 역할

```
✅ 체크해야 할 것:
  - 로직 정확성 (엣지 케이스, 경쟁 조건)
  - 설계 (책임 분리, 의존성 방향)
  - 보안 (입력 검증, SQL injection, 민감 정보 노출)
  - 테스트 커버리지 (핵심 경로 커버 여부)
  - 가독성 (미래의 팀원이 이해할 수 있는가)

❌ 스타일/포매팅:
  → Lint/Formatter가 처리. 리뷰어가 잡지 않는다.
```

---

## 2. 리뷰 코멘트 레벨 (명시적 표기)

코멘트 앞에 중요도 레벨을 붙여 혼선을 없앤다.

```
[blocker] 반드시 수정해야 Merge 가능
[major]   중요한 문제이나 협의 가능
[minor]   작은 개선사항 (리뷰이 재량)
[nit]     사소한 취향/스타일 (무시해도 됨)
[question] 이해를 위한 질문 (수정 요청 아님)
[praise]  잘된 부분 칭찬
```

```
// 예시 코멘트
[blocker] 여기서 트랜잭션 없이 두 번 DB에 쓰면 실패 시 불일치가 생깁니다.
  @Transactional 추가 또는 보상 로직이 필요합니다.

[nit] 변수명 `d`보다 `orderDate`가 의도가 명확할 것 같습니다.

[question] 여기서 Redis 캐시를 쓰는 이유가 있나요? 성능 이슈가 있었던 건지 궁금합니다.

[praise] Optional 체이닝으로 NPE 방어 깔끔하게 됐네요 👍
```

---

## 3. 리뷰이(PR 작성자)의 역할

```
PR 올리기 전 셀프 리뷰:
  □ PR 설명에 "왜 이렇게 했는지" 포함
  □ 큰 변경은 먼저 팀과 설계 논의 후 코딩
  □ 자동화 테스트 통과 확인
  □ WIP(작업 중)이면 Draft PR으로

코멘트 받은 후:
  □ 모든 [blocker]·[major] 해결 또는 반론
  □ 해결했으면 "Done" 또는 해당 커밋 링크 답글
  □ 의견 차이는 채팅/구두로 해결 후 정리 댓글
  □ 개인 감정으로 받아들이지 않기
```

---

## 4. 리뷰 사이클 목표

```
PR 생성 후 최초 리뷰: 영업일 1일 이내
리뷰 반영 후 재검토: 반나절 이내
총 사이클: 2~3일 이내 Merge 목표

→ "리뷰 전 24시간 이상 방치" 는 팀 속도 저하 신호
→ 급한 건 @멘션 또는 슬랙 알림
```

---

## 5. 리뷰하기 어려운 PR 유형

| 유형 | 문제 | 해결 |
|------|------|------|
| 1000줄+ 거대 PR | 리뷰어 집중력 한계 | 기능 단위로 쪼개기 |
| 맥락 없는 리팩터링 | 의도 파악 불가 | 설명 댓글 또는 이슈 연결 |
| WIP 코드 섞임 | 무엇이 완성인지 모름 | Draft PR 분리 |
| 오래된 브랜치 | 충돌 많음, 히스토리 복잡 | 정기 rebase |

---

## 6. LGTM 기준

```
LGTM(Looks Good To Me) 조건:
  ✅ [blocker] 코멘트 없음
  ✅ 테스트 CI 통과
  ✅ 변경 목적에 맞는 테스트 존재
  ✅ 보안 취약점 없음

LGTM 줄 수 없는 경우:
  ❌ 테스트 없는 비즈니스 로직 변경
  ❌ 하드코딩된 시크릿
  ❌ 미완성 TODO 남긴 채 Merge 요청
```
