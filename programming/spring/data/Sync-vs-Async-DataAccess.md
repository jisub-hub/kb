# 동기(Blocking) vs 비동기(Reactive) 데이터 접근

> [!summary] 한 줄 요약
> JDBC 계열(JDBC/MyBatis/JPA)은 **블로킹**(스레드당 요청), R2DBC는 **논블로킹**(이벤트 루프, 적은 스레드로 많은 동시성). "비동기가 항상 빠르다"는 **오해**다 — 워크로드에 따라 결정해야 한다.

---

## 1. 블로킹 모델 (Thread-per-Request)
```
요청 N개 → 스레드 N개 → 각 스레드가 DB 응답까지 대기(블로킹)
```
- 스레드가 I/O 동안 **놀면서 점유** → 동시성이 스레드 풀 크기에 제한.
- 단순하고 직관적, 디버깅 쉬움, 생태계 풍부.
- 대부분의 비즈니스 앱은 이 모델로 **충분**(HikariCP + 적절한 풀 크기).

## 2. 논블로킹/리액티브 모델 (Event Loop)
```
요청 N개 → 소수의 이벤트 루프 스레드가 I/O 대기 중 다른 요청 처리
```
- 스레드가 **대기하지 않음** → 적은 스레드로 높은 동시성(특히 I/O 바운드).
- 대신 **모든 구간이 논블로킹**이어야 효과 (한 군데라도 블로킹되면 무너짐).
- 복잡도·학습비용 높고, 디버깅 어렵다.

---

## 3. 비교표

| 항목 | 동기 (JDBC/JPA/MyBatis) | 비동기 ([[R2DBC]]) |
|------|------------------------|------|
| 스레드 모델 | thread-per-request | event loop |
| 스택 | Spring MVC | Spring WebFlux |
| 반환 타입 | 객체/List | Mono/Flux |
| 동시성(I/O 바운드) | 스레드 풀 한계 | 매우 높음 |
| ORM | JPA로 풍부 | 없음(단순 매핑) |
| 학습/디버깅 | 쉬움 | 어려움 |
| 생태계 성숙도 | 매우 높음 | 발전 중 |
| 트랜잭션/연관관계 | 강력 | 제한적 |

---

## 4. 선택 가이드

> [!success] 동기(블로킹)를 선택 — 대부분의 경우
> - 일반적인 CRUD/비즈니스 애플리케이션
> - 복잡한 도메인·연관관계·트랜잭션 → [[JPA-Hibernate]]
> - 복잡 SQL/튜닝 → [[MyBatis]]
> - 팀 생산성·유지보수가 중요

> [!success] 비동기(리액티브)를 선택 — 특정 조건
> - **이미 WebFlux 리액티브 스택** 전체
> - 매우 높은 동시 연결 + I/O 바운드(게이트웨이, 스트리밍, 외부 API 다중 호출)
> - 백프레셔가 필요한 데이터 스트림

> [!warning] 흔한 오해
> - "비동기 = 항상 빠름" ❌ → **CPU 바운드**나 단순 트래픽에선 이점이 없고 오히려 복잡도만 증가.
> - 리액티브 스택에 **블로킹 코드(JDBC, `.block()`) 혼입** → 이벤트 루프가 막혀 최악의 성능.
> - 동기 스택의 처리량은 보통 **커넥션 풀/스레드 풀 튜닝**([[Connection-Pool]])으로 충분히 끌어올릴 수 있다.

---

## 5. 가상 스레드(Virtual Threads, Project Loom)
- Java 21+ 가상 스레드 + Spring 6.1/Boot 3.2 → **블로킹 코드를 그대로 쓰면서** 높은 동시성.
- "리액티브의 복잡함 없이 동시성"을 노리는 흐름 → 많은 경우 R2DBC의 대안이 될 수 있다.
- JDBC/JPA 코드 + 가상 스레드 조합이 앞으로 주류 선택지 중 하나로 부상 중.

---

## 6. 📋 심층 리포트 — "I/O 바운드 + DB 통신 많으면 WebFlux 도입 가치 있다?"

> **질문**: CPU 바운드 작업이 별로 없고, DB와 주고받는 network I/O가 많다면 WebFlux를 도입할 가치가 있는가?
> **결론**: **"조건부로 맞다(partially true)."** 방향은 맞지만, 그대로 적용하면 틀리기 쉬운 **3가지 전제**가 빠져 있다.

### 6.1. 왜 "방향"은 맞나
리액티브의 이점은 **스레드가 I/O 대기로 노는 시간을 없애는 것**이다.
- CPU 바운드: 스레드가 계속 일함 → 놀지 않음 → **리액티브 이점 없음**(오히려 컨텍스트 전환 손해).
- I/O 바운드: 스레드가 DB/네트워크 응답을 **기다리며 점유** → 블로킹 모델에선 그 시간이 낭비.
→ 그래서 "CPU 바운드 적고 I/O 많다"는 **리액티브가 빛나는 전제 조건의 절반**은 맞다.

### 6.2. ⚠️ 빠진 전제 1 — "전 구간 논블로킹"이어야 한다 (가장 중요)
이게 핵심이다. **"DB 통신이 많다"의 그 DB 접근이 블로킹이면 WebFlux는 무의미하거나 해롭다.**

```
[케이스 A] WebFlux + JDBC/JPA(블로킹)   ← 흔한 착각
  Netty 이벤트 루프 스레드(보통 코어수 ~4~8개)에서 JDBC 호출
  → 이벤트 루프가 DB 응답까지 "블로킹"됨
  → 소수의 루프 스레드가 막혀 전체 처리량 붕괴 (MVC보다 더 나쁠 수 있음)

[케이스 B] WebFlux + R2DBC(논블로킹)     ← 이래야 이득
  DB 대기 중에도 이벤트 루프가 다른 요청 처리
  → 적은 스레드로 높은 동시성 달성 ✅

[케이스 C] WebFlux + JDBC + boundedElastic 격리
  Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())
  → 블로킹을 별도 스레드풀로 격리 (동작은 하지만)
  → 결국 "스레드 풀 기반"이라 MVC 대비 이점 거의 없음 + 복잡도만 증가
```

> [!danger] 핵심
> "DB I/O가 많다" → WebFlux가 답이 되려면 그 DB 드라이버가 **[[R2DBC]] 같은 논블로킹**이어야 한다.
> JPA/MyBatis를 쓸 거라면 WebFlux 도입의 실익은 거의 없다. **이 전제가 질문에서 빠진 가장 큰 함정.**

### 6.3. ⚠️ 빠진 전제 2 — "동시성이 충분히 높아야" 한다
- 리액티브 이점은 **동시 연결 수가 많을 때(수천~수만)** 두드러진다.
- 트래픽이 낮거나 중간 수준이면, 블로킹 모델 + 적절한 [[Connection-Pool]]/스레드풀로 **충분히 처리**되고 차이를 못 느낀다.
- 오히려 디버깅·러닝커브·운영 복잡도 비용이 이득을 초과한다.

### 6.4. ⚠️ 빠진 전제 3 — DB 자체가 병목이면 무의미
- WebFlux는 **애플리케이션 스레드** 효율을 올릴 뿐, **DB의 처리 한계**를 올리지 않는다.
- DB 커넥션 수·쿼리 성능이 병목이면, 앱이 논블로킹이어도 **DB 앞에서 대기**가 쌓인다.
- R2DBC도 결국 커넥션 풀 크기에 묶인다. → 캐시([[Redis]])·읽기 분리([[CQRS]])·쿼리 튜닝이 먼저일 수 있다.

### 6.5. 🔑 그래서 실무 의사결정 플로우
```
CPU 바운드가 큰가? ──Yes──► MVC (리액티브 무의미)
   │No
   ▼
동시 연결이 매우 높은가(수천+)? ──No──► MVC (+필요시 가상스레드)
   │Yes
   ▼
DB/외부호출을 전 구간 논블로킹(R2DBC/리액티브 클라이언트)으로 만들 수 있나?
   │            │No ──► MVC 권장 (WebFlux+블로킹은 함정)
   │Yes
   ▼
백프레셔/스트리밍 요구가 있거나 게이트웨이성 워크로드인가?
   │Yes ──► ✅ WebFlux 도입 가치 높음
   │No  ──► 가상 스레드(6.6) 먼저 검토 권장
```

### 6.6. 🆕 2024~ 현실: 가상 스레드가 판을 바꿨다
- Java 21 **가상 스레드** + Spring Boot 3.2(`spring.threads.virtual.enabled=true`)
- **블로킹 코드(JDBC/JPA)를 그대로 쓰면서** I/O 바운드 고동시성을 처리 → 스레드가 싸져서 "대기 점유" 문제가 거의 사라짐.
- 즉, "**I/O 바운드라 동시성이 필요하다**"는 이유만으로 WebFlux를 갈 필요가 줄었다.
- **WebFlux가 여전히 우위인 경우**: 스트리밍/백프레셔, 이벤트 기반 파이프라인, 함수형 리액티브 조합(WebClient 다중 호출 합성), 게이트웨이([[Spring-Cloud-Gateway]]).

### 6.7. 한 줄 정리
> "I/O 바운드 + DB 통신 多 → WebFlux 가치 있다"는 **R2DBC(전 구간 논블로킹) + 높은 동시성 + 스트리밍/백프레셔 요구**가 함께 성립할 때 참이다.
> JPA/MyBatis로 DB를 친다면, 같은 목표는 **MVC + 가상 스레드**로 더 적은 복잡도로 달성하는 것이 2024년 이후의 현실적 정답인 경우가 많다.
