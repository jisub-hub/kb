# 매니지드 DB(RDS) vs VM 직접 구축 (Self-Hosted)

> [!summary] 한 줄 요약
> **RDS(매니지드)**: 클라우드가 백업·패치·페일오버·모니터링을 대신 → 운영 부담↓·속도↑, 대신 비용↑·제어권↓. **VM 직접 구축**: 완전한 제어·낮은 단가·OS레벨 접근 가능, 대신 모든 운영 책임을 떠안음. "DB 운영 전문 인력이 있는가"가 핵심 갈림.

---

## 1. 무엇이 다른가 — 책임 경계 (Shared Responsibility)
```
[VM 직접 구축(EC2 위 PostgreSQL)]        [RDS / Aurora]
 OS/패치          : 내가              OS/패치          : AWS
 DB 설치/버전업   : 내가              DB 마이너 패치   : AWS(자동/창)
 백업/복구        : 내가 스크립트     백업/PITR        : AWS(자동)
 HA/페일오버      : 내가 구성         Multi-AZ 페일오버: AWS(자동)
 모니터링         : 내가 구축         CloudWatch/PI    : AWS 제공
 스토리지 확장    : 내가             오토스케일링      : AWS
 → 통제력 최대, 운영 부담 최대        → 운영 부담 최소, 통제력 제한
```

---

## 2. 항목별 상세 비교

| 항목 | RDS / Aurora (매니지드) | VM 직접 구축 (EC2 등) |
|------|------------------------|----------------------|
| **운영 부담** | 낮음 (자동화) | 높음 (전부 직접) |
| **초기 구축 속도** | 수 분 (콘솔/IaC) | 느림 (설치·튜닝) |
| **백업/PITR** | 자동 스냅샷 + 시점복구 | 직접 구성(pgBackRest 등) |
| **HA/페일오버** | Multi-AZ 자동 페일오버 | 직접(Patroni/repmgr/keepalived) |
| **읽기 확장** | Read Replica 클릭 생성 | 직접 복제 구성 |
| **패치/업그레이드** | 관리형(유지보수 창) | 직접(다운타임 계획) |
| **모니터링** | CloudWatch, Performance Insights | 직접(Prometheus/[[Metrics]]) |
| **OS/superuser 접근** | ❌ 불가(제한된 권한) | ✅ 완전 접근 |
| **확장(extension)** | AWS 허용 목록만 | ✅ 무엇이든 설치 |
| **세밀한 튜닝** | 파라미터 그룹 내 제한 | ✅ 커널/`postgresql.conf` 전부 |
| **비용 단가** | 높음(관리 프리미엄) | 낮음(인스턴스 값만) |
| **총소유비용(TCO)** | 인건비 절감 포함 시 유리할 때 많음 | 운영 인력 비용 별도 |
| **벤더 락인** | 있음(특히 Aurora) | 낮음(이식 용이) |
| **컴플라이언스** | AWS 인증 활용 | 직접 입증 |

---

## 3. RDS vs Aurora (둘 다 매니지드지만 다름)
- **RDS (PostgreSQL/MySQL)**: 표준 엔진을 그대로 매니지드. 호환성 100%.
- **Aurora**: AWS가 재설계한 스토리지 계층(분산·자동확장). **성능·확장성 우수**하지만 **락인 강함**, 비용 구조 다름.
- 표준 PG 기능/이식성 중요 → RDS, 극단적 성능/확장 → Aurora.

---

## 4. 커넥션 관리 관점 (중요)
- RDS/Aurora도 **인스턴스 메모리에 비례한 `max_connections` 한계**가 있다 → 앱 인스턴스 폭증 시 동일한 고갈 문제.
- AWS는 이를 위해 **RDS Proxy**(매니지드 [[PgBouncer]] 격)를 제공 → 서버리스/오토스케일 환경의 커넥션 폭증 흡수 + 페일오버 시 연결 유지.
- 직접 구축이면 PgBouncer를 **직접 운영**해야 함. → [[Connection-Pool]] / [[PgBouncer]]

---

## 5. 언제 무엇을 고르나

> [!success] RDS/Aurora (매니지드) 권장
> - **DBA/인프라 전담 인력이 부족**한 팀(대부분의 스타트업/일반 서비스)
> - 빠른 출시, 운영 자동화(백업/HA/패치)에 가치
> - 트래픽 변동이 크고 스케일링이 필요
> - 컴플라이언스를 클라우드 인증으로 충족하고 싶을 때

> [!success] VM 직접 구축 권장
> - **특수 확장/커널 튜닝/superuser**가 꼭 필요(매니지드가 막는 기능)
> - 매우 큰 규모에서 **단가 절감**이 인건비를 압도(전담 DBA 보유)
> - 멀티클라우드/온프레미스 이식성, 벤더 락인 회피가 전략적으로 중요
> - 규제/망분리로 클라우드 매니지드 사용 불가

> [!warning] 흔한 오해
> "직접 구축이 무조건 싸다" ❌ — 인스턴스 단가는 싸도 **백업·HA·모니터링·장애대응 인건비**까지 더한 TCO로 보면 소규모 팀은 RDS가 더 쌀 때가 많다.
