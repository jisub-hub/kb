# MVC + Virtual Threads vs WebFlux — 성능 비교 리포트

> [!summary] 한 줄 요약
> 둘 다 "적은 OS 스레드로 높은 I/O 동시성"을 달성하지만 **방식이 다르다.** 처리량(throughput)·테일 레이턴시는 **대체로 비슷한 수준**으로 수렴하고, 진짜 차이는 **프로그래밍 모델·메모리·백프레셔·생태계**에서 갈린다. "둘 중 뭐가 더 빠른가"보다 "어디서 비용을 치르나"가 핵심.

> [!warning] 벤치마크 수치 주의
> 아래 수치는 커뮤니티 벤치마크에서 **자주 관찰되는 경향**을 보여주는 **예시(illustrative)** 다. 하드웨어·드라이버·부하 패턴·튜닝에 따라 크게 달라진다. **반드시 본인 워크로드로 측정**하라.

---

## 1. 두 모델의 동작 차이

### MVC + Virtual Threads (Java 21+)
```
요청마다 가상 스레드 1개 (수백만 개 생성 가능, 힙에 저장)
가상 스레드가 블로킹 I/O 호출 → JVM이 자동으로 "언마운트"
→ 캐리어(OS) 스레드는 다른 가상 스레드를 실행 (놀지 않음)
I/O 완료되면 다시 마운트하여 재개
```
- **블로킹 코드를 그대로 작성**(JDBC/JPA OK) 하지만 OS 스레드는 안 막힌다.
- 동기적 사고방식 유지 → 코드/스택트레이스/디버깅 단순.

### WebFlux (Reactor, Event Loop)
```
소수 이벤트 루프 스레드(보통 코어수)
요청은 콜백/연산자 체인(Mono/Flux)으로 표현
I/O는 논블로킹 → 콜백으로 재개, 스레드는 다른 작업 계속
```
- **전 구간 논블로킹** 필요(R2DBC, WebClient). 블로킹 끼면 붕괴.
- 백프레셔(흐름 제어)가 **프로토콜 레벨로 내장**.

## 2. 항목별 비교

| 항목 | MVC + Virtual Threads | WebFlux |
|------|----------------------|---------|
| **처리량(I/O 바운드, 고동시성)** | 높음 | 높음 (대체로 동급) |
| **테일 레이턴시(p99)** | 양호 | 양호 (부하 매우 높을 때 약간 유리할 수 있음) |
| **요청당 메모리** | 가상스레드 스택(수 KB~) | 이벤트루프+연산자 객체(매우 작음) |
| **극단적 동시성(수십만+)** | 가능 | 약간 더 효율적인 경향 |
| **CPU 바운드** | 둘 다 이점 없음 (코어 수가 한계) | 동일 |
| **프로그래밍 모델** | 동기(쉬움) ✅ | 리액티브(어려움) |
| **디버깅/스택트레이스** | 정상적 ✅ | 끊김/난해 |
| **블로킹 라이브러리(JDBC/JPA)** | 그대로 사용 ✅ | 사용 불가(R2DBC 필요) |
| **백프레셔** | 없음(직접 제어) | 내장 ✅ |
| **스트리밍(SSE/대용량)** | 제한적 | 강력 ✅ |
| **생태계 성숙도** | 기존 그대로 ✅ | 리액티브 라이브러리 필요 |

## 3. 처리량/레이턴시 — 무엇이 결정하나

### 핵심 통찰: 동시성 모델은 "스레드 낭비"만 없앤다
두 방식 모두 **I/O 대기 중 OS 스레드를 놀리지 않는다**는 동일한 목표를 다른 방식으로 달성한다. 그래서:
- **I/O 바운드 + 고동시성**에서는 둘의 **처리량이 비슷한 수준**으로 수렴하는 경우가 많다.
- 진짜 상한은 보통 **다운스트림**(DB 커넥션 풀, 외부 API)이 결정한다 → [[Connection-Pool]].

### 예시 경향 (Spring Boot 3.2+, 외부 호출/DB 대기 위주, 고동시성 부하)
```
처리량(req/s):   MVC(플랫폼 스레드) <<  MVC+VirtualThreads ≈ WebFlux
                 (스레드풀 한계)        (둘 다 스레드 낭비 제거)

p99 레이턴시:    MVC(플랫폼) 높음(큐잉) >  MVC+VT ≈ WebFlux (낮음)
                 부하가 극단으로 가면 WebFlux가 메모리/스케줄링상 미세 우위인 사례

메모리:          WebFlux ≤ MVC+VT  (가상스레드 스택 vs 작은 연산자 객체)
```
> 즉, **"평범한 플랫폼 스레드 MVC → (가상스레드 or WebFlux)"** 로 가는 점프는 크지만, **가상스레드 ↔ WebFlux** 사이 처리량 차이는 보통 작다.

## 4. 각각이 이기는 지점

### MVC + Virtual Threads가 유리
- 기존 **블로킹 스택(JPA/MyBatis/JDBC)** 을 유지하면서 동시성만 올리고 싶을 때 ✅
- 팀 생산성·유지보수·디버깅 용이성이 중요할 때
- 표준적인 REST CRUD 서비스([[REST-API]])
- 마이그레이션 비용 최소화(설정 한 줄 수준)

### WebFlux가 유리
- **스트리밍/SSE/대용량 응답**, 백프레셔가 필요한 데이터 파이프라인
- **여러 외부 호출 합성/병렬화**(`Mono.zip`, `flatMap` 동시성 제어)
- 게이트웨이성 워크로드([[Spring-Cloud-Gateway]])
- 이미 엔드투엔드 리액티브([[R2DBC]] + WebClient) 스택
- 극단적 동시성에서 메모리 효율이 빠듯할 때

## 5. 함정 (각 모델의 성능 킬러)

### 가상 스레드 함정
```java
// ❌ Pinning: synchronized 블록 안에서 블로킹 I/O
synchronized (lock) {
    jdbcCall();        // 가상스레드가 캐리어 스레드에 "고정(pin)"되어 언마운트 불가
}                      // → 동시성 이점 상실. ReentrantLock으로 교체 권장
```
- `synchronized` + 블로킹 → **pinning** (Java 24부터 상당 부분 완화되었으나 여전히 주의).
- ThreadLocal 남용 시 메모리(가상스레드 수백만 개 × ThreadLocal).
- 커넥션 풀이 진짜 병목 → 가상스레드 무한 생성해도 DB 앞에서 대기.

### WebFlux 함정
```java
// ❌ 이벤트 루프에서 블로킹 → 전체 처리량 붕괴
public Mono<User> bad(Long id) {
    return Mono.just(jpaRepo.findById(id).get());  // JDBC 블로킹!
}
```
- 한 군데 블로킹(JDBC, `.block()`, 무거운 CPU 연산)이 이벤트 루프를 막음 → **MVC보다 느려짐**.
- `subscribeOn(boundedElastic())`로 격리하면 동작하지만 "스레드풀 기반"이라 이점 희석.
- 디버깅·러닝커브 비용이 성능 이득을 초과할 수 있음.

## 6. 의사결정 가이드

```
CPU 바운드 큼?            ──Yes──► 둘 다 무의미. 코어 확장/병렬화로 해결
   │No (I/O 바운드)
   ▼
스트리밍/백프레셔/리액티브 합성 필요?  ──Yes──► WebFlux
   │No
   ▼
기존 블로킹 스택(JPA/MyBatis) 유지?    ──Yes──► ✅ MVC + Virtual Threads (가장 실용적)
   │No / 신규 + 엔드투엔드 논블로킹 가능
   ▼
극단적 동시성 + R2DBC 가능?            ──Yes──► WebFlux도 좋은 선택
                                       ──No ──► MVC + Virtual Threads
```

> [!tip] 2024년 이후의 실무 합의(경향)
> - 대부분의 일반 서비스: **MVC + 가상 스레드**가 "복잡도 대비 성능"의 스윗스팟.
> - WebFlux는 **명확한 리액티브 요구(스트리밍·백프레셔·게이트웨이·고도 합성)** 가 있을 때 선택.
> - "단지 빠르려고" WebFlux를 도입하던 동기의 상당 부분을 가상 스레드가 흡수했다.

## 7. 활성화 방법

### MVC + Virtual Threads (Spring Boot 3.2+, Java 21+)
```yaml
spring:
  threads:
    virtual:
      enabled: true     # 톰캣 요청 처리 + @Async 등을 가상 스레드로
```
> 기존 코드 변경 거의 없음. JDBC/JPA 그대로. 단, `synchronized` 블로킹 구간 점검.

### WebFlux
```groovy
implementation 'org.springframework.boot:spring-boot-starter-webflux'
// + R2DBC / reactive 클라이언트로 전 구간 논블로킹 구성 필요
```

## 8. Virtual Threads + JPA vs WebFlux + R2DBC 결정 매트릭스

앞 절들이 "동시성 모델"을 비교했다면, 실무 결정은 거의 항상 **데이터 접근 스택과 묶여서** 내려진다. 가상 스레드의 진짜 파급력은 여기서 나온다.

### 8.1 왜 데이터 접근 관점이 트레이드오프를 바꾸나
[[Sync-vs-Async-DataAccess]]·[[MVC-vs-WebFlux]]는 "블로킹 JDBC를 이벤트 루프에서 호출하지 말라"는 **경고**에 머문다. 그 경고의 전제는 *블로킹 = 비싸다(OS 스레드 점유)* 였다.

Java 21+ 가상 스레드는 이 전제를 깬다.
- 블로킹 JDBC 호출 → 가상 스레드만 언마운트, **캐리어(OS) 스레드는 안 막힌다**.
- 즉 **"블로킹 JPA를 값싸게"** 쓸 수 있게 되었고, R2DBC로 가야 했던 동기의 상당 부분이 사라진다.
- 더 이상 "고동시성 = 논블로킹 드라이버 필수"가 아니다. **명령형 JPA 코드 + 가상 스레드**로 같은 동시성을 얻는다.

> 핵심: WebFlux+R2DBC를 정당화하던 "성능" 이유의 대부분을 VT+JPA가 흡수했다. 남는 차이는 **스트리밍·백프레셔·극단적 동시성·생태계**다.

### 8.2 결정 매트릭스 — VT+JPA(JDBC) vs WebFlux+R2DBC

| 기준 | VT + JPA/JDBC | WebFlux + R2DBC |
|------|---------------|-----------------|
| **단일 요청 지연** | 양호 (JPA 오버헤드 존재하나 무관) | 양호 (드라이버 경량) |
| **지속 동시성(steady)** | 높음 — 단, **커넥션 풀이 상한** | 높음 — 커넥션도 논블로킹 획득 |
| **극단적 동시성(수십만)** | 가상스레드 스택 메모리 누적 | 더 효율적 (작은 연산자 객체) |
| **GC 압박** | 가상스레드 스택·엔티티/영속성 컨텍스트 → 압박 큼 | 연산자 객체 위주, 상대적으로 낮음 |
| **디버깅·스택트레이스** | 정상 동기 스택 ✅ | 끊긴 스택, 리액터 연산자 추적 난해 |
| **생태계 성숙도** | JPA/Hibernate/QueryDSL 전부 사용 ✅ | R2DBC 매핑 빈약, 지연로딩·연관관계 없음 |
| **코드 복잡도** | 명령형, 트랜잭션 `@Transactional` 그대로 ✅ | `Mono`/`Flux` 체인, 선언적 트랜잭션 제약 |
| **트랜잭션 모델** | 익숙한 스레드바운드 트랜잭션 | 리액티브 트랜잭션(컨텍스트 전파 주의) |

### 8.3 VT + JPA가 이기는 경우
- 익숙한 **명령형 코드**·기존 팀 역량을 유지하고 싶다.
- **JPA 생태계**(연관관계, 지연로딩, 영속성 컨텍스트, QueryDSL, 스펙)가 핵심 자산이다 → [[../data/JPA-Hibernate]].
- 디버깅·장애분석이 잦다 → 정상 스택트레이스가 운영 비용을 크게 줄인다.
- **대부분의 CRUD·도메인 서비스** — 동시성은 가상 스레드로 충분, 데이터는 JPA로.

### 8.4 WebFlux + R2DBC가 이기는 경우
- **극단적 동시성·스트리밍·백프레셔**가 본질적 요구다(대용량 결과 스트리밍, SSE, 데이터 파이프라인).
- **이미 엔드투엔드 리액티브 스택**이라 R2DBC가 자연스럽다.
- 여러 다운스트림을 리액티브로 합성·병렬화하고 흐름 제어까지 필요하다.
- R2DBC의 빈약한 매핑·수동 연관관계 처리 비용을 감수할 명확한 이유가 있다.

### 8.5 VT 함정 — "가상 스레드면 다 해결"이 아니다
- **Pinning**: `synchronized` 블록·일부 native 호출 안에서 블로킹하면 가상 스레드가 캐리어 스레드에 **고정**되어 언마운트 불가 → 동시성 이점 상실(Java 24부터 상당 완화, 그래도 점검 대상). `ReentrantLock`으로 교체 권장.
- **커넥션 풀이 여전히 병목**: 가상 스레드는 수백만 개라도 **DB 커넥션은 한정**(HikariCP 풀 크기)이다. 수만 개의 가상 스레드가 풀 앞에서 줄을 선다 → 실제 동시성 상한은 풀이 결정 → [[Connection-Pool]]. "가상 스레드 수 ≠ 동시 DB 처리량".
- **ThreadLocal 폭증**: 요청당 가상 스레드 + 가상 스레드마다 ThreadLocal(보안 컨텍스트, MDC 등) → 수백만 개 × ThreadLocal로 메모리 폭증. `ScopedValue`(Java 21+ preview→정식화 진행) 고려.

### 8.6 결론
- **신규 프로젝트 대부분은 VT + JPA가 합리적 기본값**이다 — 명령형 단순함 + JPA 생산성 + 가상 스레드 동시성.
- 리액티브(WebFlux + R2DBC)는 **스트리밍·백프레셔·극단적 동시성처럼 명확한 이유가 있을 때** 선택한다.
- 단, "VT면 무조건 빠르다"는 환상은 금물: 진짜 상한은 **커넥션 풀과 다운스트림**이며, pinning·ThreadLocal을 점검해야 한다.
