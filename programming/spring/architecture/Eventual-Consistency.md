# Eventual Consistency (결과적 일관성)

> [!summary] 한 줄 요약
> 모든 노드가 **항상** 같은 값을 보이지는 않지만, **더 이상 갱신이 없다면 결국(eventually) 모두 같은 값으로 수렴**한다는 분산 시스템의 일관성 모델.

---

## 1. 강한 일관성 vs 결과적 일관성

| 구분 | 강한 일관성 (Strong) | 결과적 일관성 (Eventual) |
|------|----------------------|--------------------------|
| 읽기 보장 | 쓰기 직후 모든 읽기가 **최신값** | 일정 시간 동안 **옛값**이 보일 수 있음 |
| 동기화 | 동기 (커밋까지 대기) | 비동기 (전파 지연 허용) |
| 가용성 | 낮음 (조율 비용) | 높음 |
| 지연(latency) | 높음 | 낮음 |
| 예시 | 단일 RDB, 2PC, 분산 락 | DNS, S3, 캐시, 이벤트 기반 MSA |

```
[강한 일관성]  write ──► (모든 복제본 동기 반영) ──► 이후 read는 항상 최신
[결과적 일관성] write ──► 비동기 전파 ──► 수렴 전 read는 옛값 가능 ──► 결국 일치
```

---

## 2. CAP 정리와의 관계

CAP: 네트워크 **분할(Partition)** 이 발생했을 때 **일관성(C)** 과 **가용성(A)** 중 하나만 택할 수 있다.

```
P(분할)은 분산 시스템에서 회피 불가 → 실질 선택은 C vs A
  CP 시스템: 일관성 우선 → 분할 시 일부 요청 거부 (예: ZooKeeper, etcd)
  AP 시스템: 가용성 우선 → 분할 중에도 응답, 나중에 수렴 → 결과적 일관성 (예: Cassandra, DynamoDB)
```

> 결과적 일관성은 **AP 진영의 일관성 모델**이다. "분할 중에도 받아주되, 일관성은 나중에 맞춘다."

> [!tip] PACELC
> CAP의 확장. "분할(P)이면 A vs C, **그 외(Else)** 엔 지연(L) vs 일관성(C)." 평상시에도 강한 일관성은 **지연을 대가로** 한다는 점을 드러낸다.

## 3. ACID vs BASE

| ACID (전통 RDB) | BASE (분산 시스템) |
|-----------------|--------------------|
| **A**tomicity | **B**asically **A**vailable (기본적 가용) |
| **C**onsistency | **S**oft state (상태가 시간에 따라 변할 수 있음) |
| **I**solation | **E**ventually consistent (결국 일관) |
| **D**urability | |

- ACID는 **강한 일관성**을, BASE는 **가용성과 확장성**을 우선한다.
- 둘은 스펙트럼의 양극단이며, 시스템마다 적절한 지점을 택한다.

---

## 4. 왜 MSA에서 필요한가

[[MSA]]에서는 **서비스마다 DB가 분리(Database per Service)** 된다. 여러 서비스에 걸친 변경을 하나의 ACID 트랜잭션으로 묶을 수 없다.

```
주문 서비스 DB     결제 서비스 DB     재고 서비스 DB
     │                  │                  │
     └──── 하나의 트랜잭션으로 묶을 수 없음 ────┘
```

- 2PC(2단계 커밋)는 가용성·성능·잠금 문제로 실무에서 기피.
- 대신 **결과적 일관성을 받아들이고**, 각 서비스의 로컬 트랜잭션을 비동기로 이어 붙인다. → [[Saga-Pattern]]
- [[CQRS]] 의 읽기 모델 분리도 이 트레이드오프 위에 선다(쓰기 후 읽기 모델 반영까지의 지연).

---

## 5. 결과적 일관성을 달성하는 패턴

- **이벤트 전파(비동기 메시징)**: 상태 변경을 이벤트로 발행, 구독 서비스가 자기 데이터를 갱신. → [[Kafka]]
- **CDC (Change Data Capture)**: DB 변경 로그(WAL/binlog)를 캡처해 전파 (Debezium).
- **Outbox 패턴**: 비즈니스 데이터와 이벤트를 같은 로컬 트랜잭션에 기록 → 발행 원자성 보장. → [[Outbox-Pattern]]
- **읽기 모델 프로젝션**: 이벤트로 비정규화 뷰를 갱신([[CQRS]], [[Event-Sourcing]]).
- **백그라운드 동기화/리컨실리에이션**: 주기적으로 불일치를 탐지·교정.

```java
// 쓰기 커밋 후, 다른 서비스/읽기 모델로 비동기 전파
@TransactionalEventListener(phase = AFTER_COMMIT)
public void on(OrderPlacedEvent e) {
    streamBridge.send("order-events", e);   // 구독자들이 결과적으로 수렴
}
```

## 6. 일관성 윈도우 (수렴 시간)

**일관성 윈도우(consistency window)**: 쓰기가 모든 복제본에 반영되어 일관성이 회복되기까지의 시간.

```
write ──┬─────────────────────┬──► 시간
        │←── 불일치 가능 구간 ──→│
        쓰기 시점            수렴 완료
        (= 일관성 윈도우)
```

- 보통 수 ms ~ 수 초. 부하·네트워크·복제 토폴로지에 따라 변동.
- 이 윈도우 동안 **stale read(옛값 읽기)** 가 발생할 수 있다 → 설계로 흡수해야 한다.

---

## 7. 일관성 모델의 종류

순수 결과적 일관성은 "언젠가 수렴"만 보장하므로, 실무에선 더 강한 **세션 보장(client-centric)** 을 얹는다.

| 모델 | 보장 내용 |
|------|-----------|
| **Read-your-writes** | 내가 쓴 값은 내가 즉시 다시 읽을 수 있다 |
| **Monotonic reads** | 한 번 본 값보다 더 옛값이 나중에 보이지 않는다 |
| **Monotonic writes** | 같은 클라이언트의 쓰기는 발생 순서대로 반영된다 |
| **Consistent prefix** | 쓰기 순서의 일관된 접두사를 본다(중간 건너뛰기 없음) |
| **Causal consistency** | 인과 관계가 있는 연산의 순서는 보존된다 |

> [!tip] read-your-writes 흔한 구현
> 방금 쓴 사용자는 **Primary(쓰기 노드)에서 읽도록** 라우팅하거나, 갱신 결과를 응답에 담아 **로컬에서 낙관적으로 갱신**한다. "내가 쓴 건 안 보이면 버그처럼 느껴진다"를 막는다.

---

## 8. 사용자 경험(UX) 대응

stale read는 기술 문제이자 **UX 문제**다. 사용자가 "방금 만든 게 안 보임"을 만나지 않게 한다.

- **낙관적 UI(Optimistic UI)**: 서버 확정 전, 성공을 가정하고 화면을 먼저 갱신.
- **상태 표시**: `PENDING` / `처리 중` 같은 중간 상태를 명시적으로 노출 (→ [[Saga-Pattern]]의 Semantic Lock과 동일 정신).
- **폴링/푸시**: 수렴 후 갱신을 폴링하거나 WebSocket/SSE로 알림.
- **read-your-writes 라우팅**: 작성자 본인에겐 즉시 최신값을 보여줌(7절).

```
[주문 완료] 클릭
  → UI 즉시 "주문 접수됨 (처리 중)" 표시  (낙관적 + 상태 명시)
  → 백그라운드에서 결제/재고 사가 진행
  → 완료 이벤트 수신 시 "주문 확정"으로 갱신
```

## 9. 멱등성 · 순서 보장과의 관계

결과적 일관성은 **비동기 메시징** 위에서 동작하므로, 메시징의 두 난제를 함께 다뤄야 한다.

- **멱등성(Idempotency)**: at-least-once 전달로 메시지가 **중복**될 수 있다. 같은 이벤트를 두 번 처리해도 결과가 같도록 → 처리된 메시지 ID 기록([[Inbox-Pattern]]).
- **순서 보장(Ordering)**: 전파 순서가 뒤바뀌면 잘못 수렴할 수 있다. → 파티션 키(예: aggregateId)로 같은 엔티티 이벤트를 한 파티션에 모아 순서 유지([[Kafka]]), 또는 이벤트에 버전/시퀀스를 실어 역전 무시.

> 멱등성·순서 보장이 없으면 "결국 일관"이 아니라 "결국 **틀린 값으로** 수렴"할 수 있다.

---

## 10. 언제 강한 일관성이 필요한가

> [!warning] 모든 곳에 결과적 일관성을 적용하지 말 것
> 도메인이 stale read를 **절대 허용하지 못하는** 지점이 있다.

| 강한 일관성 필요 | 결과적 일관성 허용 |
|------------------|--------------------|
| 결제·잔액 차감 (이중 인출 방지) | 좋아요/조회수 카운트 |
| 재고의 마지막 1개 (오버셀 방지) | 추천·랭킹 |
| 계좌 이체, 정산 | 알림/피드 타임라인 |
| 고유성 제약(중복 가입) | 검색 인덱스 |

- 결제·재고처럼 **금전적/법적 정합성**이 걸린 곳은 강한 일관성(단일 트랜잭션, 분산 락, 또는 Saga의 Semantic Lock + 보상)으로 보호.
- 카운터·피드처럼 **약간의 지연이 무해**한 곳은 결과적 일관성으로 가용성·성능을 얻는다.

> 실무 원칙: **"기본은 결과적 일관성, 정합성이 critical한 경계만 강한 일관성"** 으로 선택적으로 강화한다.
