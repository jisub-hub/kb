# 공공망 환경 보안 아키텍처 — 외부 노출 최소화

> [!summary] 한 줄 요약
> 공공망(행정망·업무망)에서 운영하는 프로젝트는 **프론트엔드 포함 모든 서버를 사설망 서브넷에 배치**하고, 외부 접근은 오직 Load Balancer 단 하나만 공개한다. 이를 통해 공격 표면을 최소화(Minimize Attack Surface)한다.

---

## 1. 왜 프론트엔드도 Private Subnet에?

일반적인 3-Tier에서는 Web(Nginx)을 Public Subnet에 두지만, **공공·금융·의료 등 보안 요구 사항이 높은 환경**에서는 다르다.

**Public Subnet에 서버를 두면:**
- 서버 자체가 공인 IP를 가짐 → 스캔·브루트포스 공격에 직접 노출
- ACG/보안그룹을 잘못 설정하면 포트가 인터넷에 직접 오픈될 수 있음
- 규정(공공기관 정보보안 지침, ISMS 등)에서 내부 서버의 직접 인터넷 노출 금지

**모든 서버를 Private Subnet에 두면:**
- 서버에 공인 IP 없음 → 인터넷에서 서버 IP 자체에 직접 접근 불가
- 유일한 진입점 = Load Balancer (관리 포인트 최소화)
- LB 하나만 공인 IP 노출 → 공격 표면 최소화

## 2. 아키텍처 구조

```
인터넷
  ↓ (공인 IP 하나만 노출)
[Load Balancer]  ← Public Subnet, 공인 IP, SSL Termination
  ↓ HTTPS → HTTP (내부망)
[Nginx / Frontend Server]  ← Private Subnet (공인 IP 없음)
  ↓
[WAS / API Server]         ← Private Subnet
  ↓
[DB Server]                ← DB Subnet (완전 격리)
```

**VCN 서브넷 구성:**
```
VCN: 10.0.0.0/16

[Public Subnet]   10.0.1.0/24   Route: 0.0.0.0/0 → IGW
  └── Load Balancer (공인 IP 보유, 인터넷 노출)

[Web Subnet]      10.0.2.0/24   Route: 0.0.0.0/0 → NAT GW
  └── Nginx / 프론트엔드 서버 (공인 IP 없음)
  └── 아웃바운드: NAT GW → 외부 라이브러리 CDN, API 등

[WAS Subnet]      10.0.3.0/24   Route: 0.0.0.0/0 → NAT GW
  └── Spring Boot API 서버

[DB Subnet]       10.0.4.0/24   Route: 내부만 (NAT GW 없음)
  └── PostgreSQL Primary/Standby
```

## 3. Load Balancer가 유일한 진입점

```
외부 클라이언트 → LB (HTTPS:443) → Nginx (HTTP:80, Private IP)
                                  → WAS (HTTP:8080, Private IP)

클라이언트는 LB의 공인 IP/도메인만 알고 있음.
내부 서버 IP는 클라이언트에서 절대 알 수 없음.
```

**LB ACG 설정:**
| 방향 | 프로토콜 | 포트 | 소스 | 설명 |
|---|---|---|---|---|
| Inbound | TCP | 443 | 0.0.0.0/0 | 인터넷 HTTPS 허용 |
| Inbound | TCP | 80 | 0.0.0.0/0 | HTTP (443으로 리다이렉트) |
| Outbound | TCP | 80 | Web Subnet | Nginx로 포워딩 |

**Web ACG 설정:**
| 방향 | 프로토콜 | 포트 | 소스 | 설명 |
|---|---|---|---|---|
| Inbound | TCP | 80 | **LB Subnet만** | LB에서만 접근 허용 |
| Outbound | TCP | 8080 | WAS Subnet | WAS로 API 요청 |
| Outbound | TCP | 443 | 0.0.0.0/0 | NPM/CDN 아웃바운드 (NAT GW 경유) |

## 4. 내부 서버 접속 방법 (운영자)

서버에 공인 IP가 없으므로 직접 SSH 접속 불가. **Bastion Host** 경유.

```
운영자 PC → (SSH, 443-VPN) → Bastion Host (Public Subnet, 공인 IP)
                                 ↓ 내부망 SSH
                           Web/WAS/DB 서버 (Private IP)
```

**Bastion Host 설정:**
```
위치: Public Subnet (또는 Management Subnet /28 소규모)
공인 IP: 탄력적 IP (고정)
ACG: SSH(22) ← 운영자 VPN IP 또는 특정 공인 IP만
```

**또는 VPN 기반 접속 (더 안전):**
```
운영자 → VPN 접속 → 사내 IP 할당 → 내부망 서버 SSH
(공인 Bastion 필요 없음, VPN Gateway가 진입점)
```

## 5. SSL/TLS 처리

```
인터넷 → LB (HTTPS 443, TLS 종료) → 내부망 (HTTP 80)

장점:
- 인증서를 LB 1곳에서만 관리
- 내부 서버는 HTTP로 통신 → 처리 부하 감소
- 내부망은 신뢰 구간으로 간주

보안 강화 (End-to-End TLS):
인터넷 → LB (HTTPS 443) → 내부망 (HTTPS 443)
- LB에서 재암호화 → 내부 서버도 인증서 관리 필요
- 내부 패킷 스니핑까지 방어 필요한 고보안 환경에서 사용
```

### 6.1 국가사이버안전관리규정 (대통령훈령 제316호)

**적용 대상**: 국가기관, 지방자치단체, 공공기관

| 조항 | 내용 | 아키텍처 적용 |
|---|---|---|
| 제17조 (망분리) | 인터넷망과 내부업무망 물리적·논리적 분리 | 모든 서버 Private Subnet, LB만 Public |
| 제18조 (접근통제) | 정보시스템 접근 권한 최소화 | ACG/NSG 이중 방화벽, IP 화이트리스트 |
| 제19조 (암호화) | 중요 정보 암호화 전송·보관 의무 | HTTPS/TLS 1.2+, DB 연결 TLS |
| 제20조 (감사로그) | 주요 정보시스템 접근 로그 기록·보관 | LB 액세스 로그, Bastion 접속 로그 |
| 제21조 (취약점 점검) | 정기 보안취약점 점검 의무 | 분기별 ACG 룰 검토, 연 1회 침투테스트 |

### 6.2 국가 사이버보안 기본지침 (국정원, 2026.5.1 시행)

구 명칭 **"국가 정보보안 기본지침"**을 2026년 5월 1일 개정하여 명칭 변경.

**핵심 변화 — 국가 망 보안체계(N2SF) 도입**

기존 일률적 인터넷망↔내부망 물리 분리 의무 → **정보 등급 기반 보안** 체계로 전환:

| 정보 등급 | 의미 | 네트워크 정책 |
|---|---|---|
| **기밀 (C, Confidential)** | 유출 시 국가 안보 영향 | 완전 분리 망, 인터넷 접근 불가 |
| **민감 (S, Sensitive)** | 유출 시 업무·개인정보 침해 | 제어된 접근, VPN/Bastion 필수, 감사 로그 |
| **공개 (O, Open)** | 일반 공개 가능 정보 | 인터넷 접근 허용 (LB 통과 후) |

```
이 아키텍처와 N2SF 매핑:
  DB Subnet      → 기밀(C) / 민감(S) 영역  (인터넷 완전 차단)
  WAS Subnet     → 민감(S) 영역           (LB에서만 인바운드)
  Web Subnet     → 민감(S) 영역           (LB에서만 인바운드)
  Load Balancer  → 공개(O) 경계           (유일한 Public 노출 지점)
  Bastion Host   → 관리자 접근 제어 경계   (IP 제한 + 감사 로그)
```

### 6.6 ISMS-P (정보보호 및 개인정보보호 관리체계 인증)

KISA(한국인터넷진흥원) 발급. **12개 분야 64개 인증기준**.

네트워크 보안 관련 주요 인증기준:

| 분야 | 기준 번호 | 인증기준 내용 | 이 아키텍처의 충족 방법 |
|---|---|---|---|
| 접근통제 | **2.6.1** | 네트워크 접근통제 (방화벽·IDS/IPS) | LB-only Public, ACG/NSG 이중화 |
| 접근통제 | **2.6.2** | 정보시스템 접근통제 (관리자 포함) | Bastion Host / VPN 경유, IP 제한 |
| 암호화 | **2.7.1** | 암호화 정책 수립 및 적용 | HTTPS/TLS 1.2+, DB 연결 TLS |
| 암호화 | **2.7.2** | 암호키 관리 | 인증서 자동 갱신, 키 관리 정책 |
| 운영보안 | **2.9.1** | 변경관리 (보안 영향 검토) | IaC로 인프라 변경 이력 관리 |
| 운영보안 | **2.9.4** | 로그 및 접속기록 관리 | LB·Bastion·DB 접근 로그 수집 |
| 모니터링 | **2.11.1** | 보안 모니터링·침해 탐지 | 이상 트래픽 알람, 접속 이상 탐지 |

> [!info] ISMS 의무 대상 (정보통신망법 제47조)
> 연매출 100억 원 이상 또는 이용자 100만 명 이상 정보통신서비스제공자, 의료기관, 교육기관 등.  
> 공공기관은 **의무 인증 대상** (전자정부법 연계).

### 6.7 CSAP (클라우드 보안인증제)

공공기관이 민간 클라우드(AWS/NCP/KT Cloud 등) 도입 시 필수 확인 인증.  
과학기술정보통신부 고시, KISA 운영.

| 등급 | 대상 데이터 | 주요 보안 요구사항 |
|---|---|---|
| **하(下)** | 공개 가능 비중요 정보 | 기본 접근통제, 암호화, 모니터링 |
| **중(中)** | 민감 정보, 개인정보 | 망분리, 암호화, 감사로그, 취약점 점검 |
| **상(上)** | 비밀·고유식별·국가 중요 정보 | 전용 인프라, 물리 분리, 보안 검증 강화 |

> 이 아키텍처 (Private Subnet + LB + Bastion + HTTPS + 로그)는 CSAP **중(中)** 등급의 기본 기술 요건을 충족하는 구조.

### 6.8 요구사항 ↔ 아키텍처 종합 매핑

| 법령 / 규정 | 요구사항 | 아키텍처 구현 |
|---|---|---|
| 국가사이버안전관리규정 제17조 | 인터넷망·내부망 분리 | 모든 서버 Private Subnet, LB만 Public IP |
| 국가 사이버보안 기본지침 N2SF | 정보 등급별 망 분리 | DB·WAS·Web = 민감(S), LB = 공개(O) 경계 |
| 정보통신망법 제45조 | 기술적 보호조치 | NSG(서브넷) + ACG(인스턴스) 이중 방화벽 |
| 개인정보보호법 제29조·시행령 제30조 | 내부망 저장 + 암호화 | DB Subnet 격리 + TLS 통신 암호화 |
| 전자정부법 제56조 | 전자정부서비스 보안 조치 | HTTPS 강제, 인증서 자동 갱신 |
| ISMS-P 2.6.1 / 2.6.2 | 네트워크·시스템 접근통제 | Bastion Host, IP 화이트리스트, 최소권한 |
| ISMS-P 2.7.1 | 암호화 적용 | HTTPS/TLS 1.2+, LB SSL Termination |
| ISMS-P 2.9.4 / 2.11.1 | 로그 관리 및 모니터링 | LB·Bastion·DB 로그 수집·보관 (최소 1년) |
| CSAP 중(中) | 망분리 + 암호화 + 감사로그 | Private Subnet + HTTPS + 로그 파이프라인 |

## 7. 일반 서비스 vs 공공/보안 환경 비교

| 항목 | 일반 서비스 | 공공/보안 환경 |
|---|---|---|
| Web/Frontend 위치 | Public Subnet | **Private Subnet** |
| 서버 공인 IP | 있음 (Web 서버) | **없음 (LB만)** |
| 진입점 | 다수 (서버별) | **LB 단 하나** |
| 관리 접속 | 직접 SSH 또는 공인 IP | Bastion / VPN 필수 |
| 인증서 관리 | 서버별 또는 LB | **LB 중앙화** |
| 내부 통신 암호화 | 선택 | End-to-End TLS 권장 |
