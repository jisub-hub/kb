# 서브네팅 (Subnetting) & CIDR

> [!summary] 한 줄 요약
> 하나의 큰 IP 대역을 **역할별로 잘게 나누는 기술**. 서브넷을 잘 설계해야 보안 정책(ACG/NSG)을 서브넷 단위로 깔끔하게 적용할 수 있다.

---

## 1. CIDR 표기법

```
192.168.1.0 / 24
            └── 슬래시(/) 뒤 숫자 = 네트워크 비트 수
                32 - 24 = 8bit → 호스트 범위 결정
```

- **네트워크 비트**: 고정된 공통 부분 (서브넷 식별자)
- **호스트 비트**: 변동 가능한 개별 IP 주소 부분
- 슬래시 숫자가 **클수록** 범위가 좁고, **작을수록** 범위가 넓다

---

## 2. 주요 CIDR 대역표

| CIDR | 서브넷 마스크 | 사용 가능 호스트 수 | 주요 용도 |
|---|---|---|---|
| /8 | 255.0.0.0 | 16,777,214 | 대형 클래스 A 네트워크 |
| /16 | 255.255.0.0 | 65,534 | VCN/VPC 전체 대역 (권장) |
| /24 | 255.255.255.0 | 254 | 일반 서브넷 (가장 흔함) |
| /28 | 255.255.255.240 | 14 | 소규모 서브넷, 클라우드 권장 최소 |
| /32 | 255.255.255.255 | 1 (단일 IP) | ACG에서 특정 IP 1개 지정 시 |

> [!warning] 클라우드 서브넷 최소 크기
> AWS/OCI는 서브넷의 첫 번째·마지막 몇 개 IP를 예약 용도로 사용한다 (AWS: 5개 예약). /28은 실제 사용 가능 IP가 14개 → 소규모 관리 서브넷에 적합. 일반 서브넷은 **/24 이상** 권장.

---

## 3. 서브넷 마스크 계산

```
/24 예시:
  IP:   192.168.1.10  = 11000000.10101000.00000001.00001010
  Mask: 255.255.255.0 = 11111111.11111111.11111111.00000000
  AND:  192.168.1.0   = 네트워크 주소 (서브넷 식별)

  호스트 범위: 192.168.1.1 ~ 192.168.1.254
  브로드캐스트: 192.168.1.255
```

---

## 4. VCN/VPC 역할별 서브넷 분할 예시

전체 VCN 대역: `10.0.0.0/16` (65,534 IP)

```
10.0.0.0/16 (전체 VCN)
├── 10.0.1.0/24  → Public Subnet   (웹서버, LB, Bastion)
├── 10.0.2.0/24  → Private Subnet  (WAS, App Server)
├── 10.0.3.0/24  → DB Subnet       (Database, 격리)
└── 10.0.4.0/28  → Management SN   (Bastion Host 전용, 소규모)
```

**서브넷 분리 이유:**
- 보안 정책(NSG/ACG)을 서브넷 단위로 일괄 적용 가능
- Public ↔ Private 경계를 명확히 → 의도치 않은 인터넷 노출 방지
- DB 서브넷을 격리하면 WAS가 뚫려도 DB로 직접 접근 차단

---

## 5. 서브넷 설계 원칙

| 원칙 | 설명 |
|---|---|
| **역할별 분리** | Web / WAS / DB / Management 서브넷 명확히 구분 |
| **Public/Private 구분** | 인터넷 노출이 필요한 것만 Public |
| **충분한 여유** | 나중에 서브넷 확장은 어려움 → 처음부터 /24 이상 권장 |
| **CIDR 중복 금지** | VPN·피어링 연결 시 대역이 겹치면 라우팅 충돌 |
| **가용 영역(AZ) 고려** | 이중화 시 AZ별로 서브넷 쌍 생성 (예: 1a·1b 각각 /24) |
