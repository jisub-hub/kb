# 파라메트릭 지식 편집·핫스왑 (DMoE Fact Hot-swapping)

> [!summary] 한 줄 요약
> RAG(외부 검색)도 전체 파인튜닝도 아닌 **제3의 길** — 모델 가중치 안의 지식(파라메트릭 지식)을 **모듈 단위로 갈아끼우는** 방향. "분리형 MoE(DMoE)의 전문가 모듈을 핫스왑해 사실을 실시간 갱신"이 그 극단적 형태. 방향은 타당하나, 전달되는 설명은 **성숙도·'깔끔함'을 과장**하는 경향이 있어 그 함정을 분명히 한다.

> 검색 기반(비파라메트릭) 지식 주입 지도는 [[RAG-vs-Knowledge-Systems]], 전체 학습은 [[Fine-Tuning]], MoE 구조 자체는 [[Transformer-Architecture]]. 이 노트는 그 사이 빠진 **"편집 가능한 파라메트릭 지식"**.

> [!warning] 검증 한계
> "칭화대 DMoE 2026" 같은 **특정 논문의 존재·세부는 이 노트로 단정하지 않는다**(지식 컷오프 밖일 수 있음). 아래는 그 **개념·주장**을 확립된 문헌(FFN=지식저장소·MoE·ROME/MEMIT·어댑터)에 비춰 분석한 것이다.

---

## 1. 출발점 — 트랜스포머에서 "지식"은 어디 사는가

```
핵심 근거(확립됨): FFN(피드포워드) 레이어 = key-value 메모리처럼 작동
  (Geva et al. 2021, "Transformer FFN Layers Are Key-Value Memories")
  Attention = 토큰 간 관계, FFN = "사실/연상"을 저장·인출하는 곳
→ MoE의 expert는 곧 FFN이다. 그래서 "expert = 지식 모듈"이라는 발상이 나온다.
```
이 한 줄이 DMoE 핫스왑 아이디어의 **진짜 토대**다. (요약이 빠뜨린 부분)

## 2. MoE → "Disaggregated MoE(DMoE)"

```
MoE: 레이어마다 expert(FFN) 여러 개 + 라우터(gating)가 토큰별로 소수 expert만 활성
  → 파라미터는 크지만 토큰당 연산은 작다(sparse). [[Transformer-Architecture]]
Disaggregation(분리): 이미 "서빙"에서는 실재하는 개념
  - Expert Parallelism: expert를 여러 GPU/노드에 분산(DeepSeek 등)
  - Prefill/Decode 분리 서빙([[LLM-System-Performance-Analysis]])
DMoE의 새 주장: expert를 "탈착 가능한 독립 지식 모듈(SSD처럼)"로 관리하고
              base model을 끄지 않고 모듈만 교체(insert/update/delete)
```
> 분리(서빙 최적화)는 확립됐지만, **"expert를 핫스왑 가능한 지식 단위로"**는 훨씬 강한 주장이고 여기가 연구 프런티어다.

## 3. 이건 새로운 게 아니라 "지식 편집" 계보의 모듈화 버전

| 계보 | 방식 | 한계 |
|------|------|------|
| **ROME / MEMIT** | FFN 가중치를 직접 rank-one 편집해 특정 사실 주입 | 편집 수 많아지면 붕괴·ripple 실패 |
| **Knowledge Neurons** | 특정 사실에 반응하는 뉴런 식별·조작 | 지역화가 깔끔하지 않음 |
| **Adapter / LoRA 핫스왑** | 작은 어댑터를 붙였다 뗌(이미 실용) | 사실 단위가 아닌 "도메인/스타일" 단위 |
| **Memory Layers** (Meta 등) | 거대 key-value 메모리를 파라미터에 두고 조회 | 편집 인터페이스·일관성 미성숙 |
| **DMoE Fact Hot-swap** | expert 모듈을 사실 단위로 탈착 | ↓ §5의 난제들 |

→ **LoRA/어댑터 핫스왑이 현존하는 "모듈 핫스왑"의 실용형**이다. DMoE는 그 입자를 "사실 단위 expert"까지 쪼개려는 시도로 보는 게 정확하다.

## 4. 비교 — RAG vs 파인튜닝 vs 지식편집/핫스왑

| 구분 | RAG | 전체 파인튜닝 | 파라메트릭 편집·핫스왑 |
|------|-----|--------------|----------------------|
| 지식 위치 | 외부 벡터 DB(비파라메트릭) | 파라미터 전체에 동화 | **분리된 모듈/국소 가중치** |
| 업데이트 비용 | 매우 낮음(DB 갱신) | 매우 높음(전체 연산) | 낮음(해당 모듈만) — *주장* |
| 추론 비용 | 컨텍스트↑로 증가 | 낮음 | 낮음(필요 모듈만) |
| 삭제/수정 | 쉬움(문서 삭제) | 사실상 불가 | 쉬움 — *주장, §5 반론* |
| 출처 추적 | 명확(문서 인용) | 불가 | 모듈=출처로 가능 — 매력적 |
| 성숙도 | 프로덕션 표준 | 표준 | **연구 단계** |

→ 비파라메트릭(RAG)과 파라메트릭(편집)은 **대체가 아니라 보완**. 결정 지도는 [[RAG-vs-Knowledge-Systems]].

## 5. ⚠️ 왜 어려운가 — 요약이 과장한 지점들

> [!danger] "모듈만 빼면 사실이 깔끔히 지워진다"는 가정이 가장 약하다
> 지식은 **국소화되어 있지 않다**. 분산·중첩(superposition)·다의(polysemantic)로 저장돼, "한 expert = 한 사실"이 성립하지 않는 경우가 많다.

```
① 지역화(localization) 문제:
   특정 사실이 한 모듈에 깔끔히 모여 있지 않음 → 빼도 잔여 지식이 다른 경로로 남음
   (해석가능성 연구: 기능은 여러 뉴런/레이어에 분산)
② 라우터 문제:
   MoE 라우터는 expert들과 "공동 학습"됨. 새 expert를 꽂아도
   라우터가 올바른 토큰을 그쪽으로 보낸다는 보장이 없음 → 라우터도 갱신 필요
③ 사실 ≠ 능력 분리 불가:
   FFN엔 사실·언어능력·추론이 뒤섞임 → 사실만 도려내면 능력도 손상 위험
④ ripple effect(일관성):
   "A의 수도는 B" 하나 고치면, 그로부터 함의되는 사실들도 같이 갱신돼야 함
   → MEMIT 계열에서도 미해결(편집은 됐는데 추론 일관성 깨짐)
⑤ 삭제(unlearning)는 오히려 난제:
   "모듈 탈착 = 삭제"는 과장. 잔여 지식·우회 추론으로 사실이 살아남음
   (machine unlearning은 그 자체로 미해결 연구 영역)
```

## 6. 그럼에도 매력적인 이유 (제대로 된 동기)

```
- 업데이트 비용: 전체 재학습 없이 국소 갱신 → 비용·탄소·다운타임↓
- 거버넌스/출처: 모듈 = 특정 출처·시점 → "이 사실은 어디서 왔나" 추적·롤백 가능
   (RAG의 인용 강점을 파라메트릭에서 흉내)
- RAG의 빈틈: 컨텍스트 오버헤드·검색 지연 없이 "내장 지식"으로 즉답
- 환각 대응: 틀린 사실을 "고치거나 제거"하는 인터페이스 — RAG는 내장 환각을 못 고침
→ RAG(빠른 갱신)와 파인튜닝(내장·빠른 추론) 사이의 빈틈을 노리는 게 본질
```

## 7. 정직한 평가 (2026-06 기준)

```
방향성: 타당하고 활발한 연구 영역(지식 편집·모듈러 모델·MoE 분리)
성숙도: 프로덕션 표준 아님. 데모·벤치 수준. RAG가 여전히 실무 1선.
전달 요약의 톤("쉽다/해결")  → 과장. §5의 지역화·라우터·일관성·unlearning이 미해결.
실무 권고: 지금 최신성·삭제가 필요하면 RAG([[RAG]]) + 거버넌스.
          파라메트릭 편집은 "주시할 프런티어"로 두되, 단일 모듈=단일 사실 가정을 의심.
```
