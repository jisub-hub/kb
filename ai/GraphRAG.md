# GraphRAG (그래프 기반 검색 증강 생성)

> [!summary] 한 줄 요약
> 문서를 **지식 그래프(엔티티·관계)로 구조화**하고, 그래프 **커뮤니티 요약**을 활용해 검색하는 RAG. 일반 벡터 RAG가 약한 **"전체를 아우르는 질문(global/테마형)"과 다중 홉 관계 추론**에 강하다. 대신 **인덱싱 비용(LLM 호출)이 크고** 단순 fact lookup엔 과하다. (Microsoft GraphRAG가 대표 구현)

> RAG 전체 지도에서의 위치는 [[RAG-vs-Knowledge-Systems]] §9, 기본 RAG는 [[RAG]], 임베딩은 [[Embedding]]. 이 노트는 **파이프라인·트레이드오프 심화**.

---

## 1. 왜 — 벡터 RAG가 못 하는 것

```
일반 벡터 RAG: 질문과 의미 유사한 청크 top-k 검색 → 그걸로 답
  잘함: "X의 정의는?" 같은 국소적 fact lookup
  약함:
    ① Global/테마 질문 — "이 1,000개 문서의 핵심 주제 5개는?" (어느 청크도 답을 안 담음)
    ② 다중 홉 관계 — "A와 C를 잇는 인물은?" (A·C는 다른 청크, 연결을 못 봄)
    ③ 흩어진 근거 종합 — 답이 여러 문서에 조각나 있을 때
→ 벡터 유사도는 "조각"을 찾지 "전체 구조"나 "연결"을 모른다.
```

## 2. 파이프라인 — 인덱싱(비쌈) + 쿼리

### 2.1 인덱싱 (오프라인, LLM 다회 호출)

```
문서 → 청크 → [LLM] 엔티티·관계 추출(triple: 주어-관계-목적어)
            → 지식 그래프 구축(노드=엔티티, 엣지=관계)
            → 커뮤니티 탐지(Leiden, 계층적) — 밀집 연결된 노드 묶음
            → [LLM] 각 커뮤니티를 요약(community report) — 계층별로
결과: 그래프 + 계층적 커뮤니티 요약들
```
- **Leiden**: 그래프를 계층적 커뮤니티로 분할(서브주제 → 상위주제). 각 레벨에 요약 생성.
- ⚠️ 여기서 LLM이 (엔티티 추출 + 커뮤니티 요약)에 **대량 호출**된다 → 비용·시간의 핵심.

### 2.2 쿼리 — Global vs Local

| 모드 | 동작 | 적합 질문 |
|------|------|----------|
| **Global search** | 커뮤니티 요약들을 **map-reduce**로 종합(각 요약에 부분답 → 통합) | "전체 주제/경향은?" 요약·테마형 |
| **Local search** | 특정 **엔티티 이웃 그래프 + 관련 텍스트** 수집 | "X에 대해/X와 Y 관계는?" 구체형 |

```
Global: 모든(또는 상위) 커뮤니티 요약을 훑어 "전역 통찰" 합성 → 토큰 많이 씀
Local : 질문 엔티티를 그래프에서 찾아 이웃·관계·원문을 모아 답 → 벡터 RAG와 비슷하나 관계 포함
```

## 3. 벡터 RAG vs GraphRAG

| 구분 | 벡터 RAG | GraphRAG |
|------|----------|----------|
| 검색 단위 | 청크(텍스트 조각) | 엔티티·관계·커뮤니티 요약 |
| 강점 | 국소 fact·저비용·빠름 | 전역 요약·관계 추론·근거 종합 |
| 인덱싱 비용 | 낮음(임베딩만) | **높음**(LLM 추출+요약 다회) |
| 갱신 | 쉬움(청크 추가) | **어려움**(그래프·커뮤니티 재계산) |
| 설명가능성 | 출처 청크 | 엔티티·관계 경로(더 구조적) |
| 지연 | 낮음 | Global은 높음(map-reduce) |

## 4. ⚠️ 함정·트레이드오프

```
① 인덱싱 비용 폭발: 엔티티 추출+커뮤니티 요약 = 코퍼스 전체에 LLM 다회 → 비싸고 느림
   → 큰 코퍼스에 무턱대고 적용 금지. ROI 먼저 따지기.
② 그래프 품질 = 추출 품질: LLM 엔티티/관계 추출이 틀리면 그래프가 오염 → 답도 오염
③ 갱신(증분) 난제: 문서 추가 시 그래프·커뮤니티를 어디까지 재계산? — 운영 부담
④ 과한 엔지니어링: 질문이 대부분 국소 fact면 벡터 RAG로 충분. GraphRAG는 오버킬
⑤ Global search 토큰 비용: 커뮤니티 요약을 많이 훑어 답당 토큰↑
```
> 핵심 판단: **"질문이 전역적·관계적인가?"** 아니면 벡터 RAG가 더 싸고 빠르다.

## 5. 변종·구현

```
Microsoft GraphRAG : 원조(엔티티 추출→Leiden 커뮤니티→global/local). 풀 파이프라인·무거움
LightRAG           : 경량화 — 그래프+벡터 듀얼, 증분 갱신 개선
nano-graphrag      : 최소 구현(학습·실험용)
Neo4j + LLM        : 그래프 DB 위에 직접 구축(Cypher 쿼리·운영 성숙)
하이브리드         : 벡터 검색 + 그래프 확장(엔티티 이웃 보강) 결합이 실무 흔함
```

## 6. 언제 쓰나 (결정)

```
GraphRAG가 값어치: 
  - 코퍼스 전체 요약·테마 추출("주요 리스크 5개")
  - 다중 홉 관계·연결 분석(수사·조사·문헌 네트워크)
  - 설명가능한 관계 경로가 중요할 때
벡터 RAG로 충분(GraphRAG 과함):
  - 대부분의 Q&A·fact lookup·문서 챗봇
  - 빠른 갱신·낮은 비용이 우선
실무: 벡터 RAG로 시작 → 전역/관계 질문이 실제 병목일 때만 그래프 도입(또는 하이브리드)
```
