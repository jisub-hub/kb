# 객체 탐지 평가 지표 — mAP, IoU, Precision/Recall

> [!summary] 한 줄 요약
> 객체 탐지는 분류와 달리 **"무엇을(클래스) + 어디에(박스)"** 둘 다 맞춰야 한다. 그래서 IoU(박스 겹침)로 정탐을 판정하고, Precision-Recall 곡선의 면적인 AP를 클래스 평균낸 **mAP**가 핵심 지표다. `mAP@50`은 관대(찾았는가), `mAP@50-95`는 엄격(박스까지 정밀한가).

---

## 1. 왜 분류 정확도로는 부족한가

```
분류(Classification): "이 이미지는 고양이다" → 맞다/틀리다 (accuracy)
탐지(Detection):      "여기(박스)에 고양이가 있다"
   → 클래스도 맞아야 하고, 박스 위치/크기도 맞아야 함
   → 단순 accuracy로 측정 불가 → IoU + AP 기반 mAP 사용
```

탐지 결과 한 건 = `[클래스, 신뢰도(confidence), 박스 좌표]`. 이걸 정답(ground truth)과 대조해 채점한다.

---

## 2. IoU — 박스가 얼마나 겹치나

```
IoU (Intersection over Union)

       예측 ∩ 정답 (교집합 면적)
IoU = ───────────────────────────   ∈ [0, 1]
       예측 ∪ 정답 (합집합 면적)

IoU = 1.0 : 완벽히 일치
IoU = 0.5 : 절반 정도 겹침
IoU = 0   : 안 겹침
```

```
┌──────────┐
│ 정답     │
│   ┌──────┼───┐
│   │ 겹침 │   │   IoU = 겹침 / (정답+예측-겹침)
└───┼──────┘   │
    │  예측    │
    └──────────┘
```

**IoU 임계값(threshold)**으로 "이 예측을 맞다고 칠지" 결정한다. 임계 0.5면 IoU ≥ 0.5인 예측만 정탐.

---

## 3. TP / FP / FN — 탐지 결과 분류

```
TP (True Positive)  : 정답과 IoU ≥ 임계 + 클래스 일치 → 제대로 맞춤
FP (False Positive) : 예측했지만 정답 없음 / IoU 미달 / 클래스 틀림 → 오탐
FN (False Negative) : 정답을 못 찾음 → 미탐
(TN은 탐지에선 의미 없음 — 배경은 무한)

주의: 한 정답에 여러 예측이 겹치면 신뢰도 1위만 TP, 나머지는 FP (중복 패널티)
```

---

## 4. Precision & Recall

```
Precision(정밀도) = TP / (TP + FP)
  "내가 탐지한 것 중 진짜 비율" → 오탐이 많으면 ↓

Recall(재현율)    = TP / (TP + FN)
  "실제 객체 중 내가 찾은 비율" → 놓치면 ↓
```

```
trade-off:
  신뢰도 임계 ↑ → 확실한 것만 예측 → Precision↑ Recall↓
  신뢰도 임계 ↓ → 많이 예측 → Recall↑ Precision↓
→ 둘은 반비례. 그래서 "곡선"으로 본다 (다음 절)
```

---

## 5. AP — Precision-Recall 곡선 아래 면적

한 클래스에 대해, 신뢰도 임계를 높은 값→낮은 값으로 내리며 (Precision, Recall) 점을 찍으면 **PR 곡선**이 그려진다.

```
Precision
  1.0 ┤■■■■■___
      │       ‾‾‾■___
      │            ‾‾■__
  0.5 ┤               ‾‾■_
      │                  ‾■
  0.0 └────────────────────── Recall
      0                    1.0

AP (Average Precision) = PR 곡선 아래 면적 (0~1)
  → Precision과 Recall을 모두 높게 유지할수록 면적↑ = 좋은 모델
  (실제 계산: 101-point 보간 등으로 면적 근사 — COCO 방식)
```

> AP 하나가 "그 클래스를 얼마나 잘 탐지하나"를 0~1로 요약한다.

---

## 6. mAP — 클래스 평균

```
mAP (mean Average Precision) = 모든 클래스 AP의 평균

mAP = (AP_굴착기 + AP_덤프트럭 + AP_크레인 + ...) / 클래스 수
```

여기까지가 "한 IoU 임계"에서의 mAP. 그런데 IoU 임계를 얼마로 잡느냐에 따라 두 가지 주요 지표가 나뉜다.

---

## 7. mAP@50 vs mAP@50-95 ⭐

| 지표 | IoU 기준 | 성격 | 의미 |
|------|---------|------|------|
| **mAP@50** (mAP@0.5) | 0.5 한 점 | 관대 | "객체를 찾았는가" |
| **mAP@75** (mAP@0.75) | 0.75 한 점 | 엄격 | 박스 정밀도 강조 |
| **mAP@50-95** (mAP@[.5:.95]) | 0.5~0.95 10단계 평균 | 종합 | **COCO 표준 주지표** |

```
mAP@50-95 계산:
  IoU 임계 = 0.50, 0.55, 0.60, ..., 0.95 (0.05 간격, 10개)
  각 임계에서 mAP 계산 → 10개를 평균
  → 박스가 정밀할수록(높은 IoU에서도 TP) 점수 유지 → 엄격
```

### 직관 — 같은 모델, 두 점수

```
mAP@50 = 0.85,  mAP@50-95 = 0.55   (흔한 패턴)

읽는 법:
  @50 높음   → "객체는 잘 찾는다"
  @50-95 낮음 → "박스 위치/크기가 살짝 어긋난다"
  @50-95 ≤ @50 항상 성립 (더 엄격하니까)

격차가 크면 → 탐지는 되는데 localization(박스 정확도)이 약함
  → 라벨 품질, 더 큰 모델, DFL(박스 회귀) 점검
```

---

## 8. 크기별 mAP — 작은/큰 객체 진단

COCO는 객체 면적별로도 mAP를 분해한다. 이게 실무 디버깅의 핵심.

| 지표 | 대상 | 낮을 때 원인·대응 |
|------|------|------------------|
| mAP_small | 작은 객체(<32²px) | 해상도 부족 → imgsz↑, SAHI, P2 head |
| mAP_medium | 중간(32²~96²) | — |
| mAP_large | 큰 객체(>96²px) | 데이터 스케일 편향 → 큰 객체 데이터·scale 증강 |

```
중장비 현장 케이스(→ [[YOLO-Finetuning]] 11절):
  mAP_small만 낮음  = A 문제 (멀리 있는 작은 표적, 해상도/SAHI)
  mAP_large만 낮음  = B 문제 (가까운 큰 표적 학습 데이터 부재)
  → 전체 mAP만 보면 원인을 못 찾는다. 반드시 크기별로 분해.
```

---

## 9. 임계값 선택 — 운영 관점

```
mAP는 모델 "비교·평가" 지표. 실제 배포 땐 운영 임계를 따로 정한다:

confidence threshold (신뢰도 컷)
  ↑ : 오탐↓ 미탐↑ (정밀 중시 — 오경보 비용 클 때)
  ↓ : 미탐↓ 오탐↑ (재현 중시 — 놓치면 안 될 때, 예: 안전·결함 검출)

IoU threshold (NMS·평가용)
  보통 NMS 0.45~0.5, 평가 0.5~0.95

안전 도메인(안전모 미착용 검출 등)은 Recall 우선 → 신뢰도 컷 낮춤
  (오탐은 사람이 거르면 되지만, 미탐은 사고로 직결)
```

---

## 10. 정리

```
IoU       : 박스가 얼마나 겹치나 (정탐 판정 기준)
Precision : 탐지한 것 중 맞은 비율 (오탐 척도)
Recall    : 실제 중 찾은 비율 (미탐 척도)
AP        : 한 클래스의 PR 곡선 아래 면적
mAP       : 클래스 AP 평균
mAP@50    : IoU 0.5 — "찾았는가" (관대, 빠른 비교)
mAP@50-95 : IoU 0.5~0.95 평균 — "박스까지 정밀한가" (엄격, 주지표)
크기별 mAP : small/medium/large로 분해 → 원인 진단
```

> 학습 평가의 1차 지표는 **mAP@50-95**. 단, **@50과 @50-95를 함께** 보고(격차=localization 문제), 작은 객체 이슈는 **크기별 mAP**로 분해해 원인을 짚는다. 운영 임계(confidence)는 도메인의 오탐/미탐 비용으로 별도 결정한다.
