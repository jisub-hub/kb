# RAG 실패 디버깅 플레이북 — 단계별 책임 귀속

> [!summary] 한 줄 요약
> "RAG가 틀린 답을 줬다"는 증상 하나로는 **어디를 고쳐야 할지 모른다**. RAG 실패는 4단계 중 하나에 귀속된다 — ① **검색 누락**(정답 청크가 후보에 없음) ② **순위**(검색됐으나 하위) ③ **컨텍스트 누락**(들어갔으나 잘림/무시) ④ **생성 실패**(근거가 있는데 답이 틀림). 핵심 기법은 **gold chunk 강제 주입**으로 검색층과 생성층을 분리해 책임을 이분(二分)하는 것. 집계 점수(RAGAS)가 아니라 *단일 오답을 단계로 가르는* 절차다.

> 집계 측정은 [[Evaluation]](RAGAS·드리프트), grounding 검증은 [[Citation-Attribution]], 속도 분해는 [[RAG-Latency-Reality]]. 이 노트는 **"이 한 답이 *왜* 틀렸나"를 단계로 귀속**하는 진단 관점.

---

## 1. 왜 단계 귀속이 필요한가

```
RAG = Retrieve → Rerank → (컨텍스트 조립) → Generate
  4단계 어디서든 답이 망가질 수 있고, 증상은 똑같이 "틀린 답"으로 보인다
집계 점수의 한계:
  RAGAS·Recall@k는 "평균적으로 어느 정도냐"는 알려주나
  → 사용자가 캡처해 보낸 "바로 이 오답"의 원인은 못 짚음
오진 비용: 검색 문제인데 프롬프트를 고치면 → 영원히 안 고쳐짐
원칙: 증상(틀린 답)에서 출발해 단계로 "이분 탐색"한다.
```

## 2. 실패의 4단계 분해 ⭐

| # | 단계 | 실패 정의 | 책임 노트(고칠 곳) |
|---|------|-----------|--------------------|
| ① | **검색(recall)** | 정답을 담은 청크가 top-N 후보에 **아예 없음** | [[Chunking-Strategies]]·[[Embedding]]·[[Query-Transformation]]·[[RAG-Ingestion-Pipeline]] |
| ② | **순위(rank)** | 정답 청크가 검색됐으나 **하위**라 top-k에서 잘림 | [[Reranking-Retrieval]] |
| ③ | **컨텍스트** | top-k엔 들어갔으나 **프롬프트에서 잘림/위치 묻힘**(예산 초과·lost-in-middle) | 컨텍스트 조립·[[Prompt-Engineering]] |
| ④ | **생성(grounding)** | 근거가 컨텍스트에 **있는데도** 답이 틀림(무시·환각·오해) | [[Citation-Attribution]]·모델/프롬프트 |

```
①②③ = "검색층" 문제(정답이 모델 눈앞에 도달 못 함)
④   = "생성층" 문제(도달했는데 못 씀)
→ 이 경계를 가르는 단 하나의 테스트가 §3.
```

## 3. 핵심 기법 — gold chunk 강제 주입 ⭐

```
질문 1개 + 사람이 아는 "정답이 든 청크(gold)"를 준비.
검색을 우회하고 gold chunk를 컨텍스트에 직접 넣어 답을 재생성:

  답이 맞아진다  → 생성층은 정상. 문제는 검색층(①②③). 검색을 파라.
  여전히 틀리다  → 생성층(④) 문제. 근거가 있어도 못 쓴다 → 프롬프트·모델·grounding.

이 한 번의 분기가 "검색 vs 생성"을 가른다(전체 디버깅의 절반).
```

> [!tip] 검색층 안에서 다시 이분
> 검색층(①②③)으로 좁혀졌으면 — gold chunk가 **검색 후보(top-N)에 있는지** 먼저 본다.
> - 후보에 **없다** → ① recall. 청킹/임베딩/쿼리변환(같은 의미인데 표현이 다른가? HyDE·동의어).
> - 후보엔 **있는데 top-k 밖** → ② 순위. 리랭커 추가/교체([[Reranking-Retrieval]]).
> - top-k엔 **있는데 답에 안 쓰임** → ③ 컨텍스트(잘림·순서) 또는 ④ 생성. 컨텍스트 길이·위치를 확인.

## 4. 단계별 진단 신호

```
① recall 의심 신호
   - 질문 용어와 문서 용어가 어휘적으로 다름(약어·동의어·다국어)
   - 정답이 표/이미지/스캔에 있음(파싱 단계 유실 → [[RAG-Ingestion-Pipeline]])
   - 청크가 너무 작아 맥락 끊김 / 너무 커 희석([[Chunking-Strategies]])
   진단: BM25 단독·dense 단독으로 각각 검색해 비교(하이브리드 누락 판별)

② 순위 의심 신호
   - top-50엔 gold가 있는데 top-5엔 없음 → bi-encoder 점수만으론 부족
   진단: cross-encoder 리랭커를 끼우고 gold 순위 변화 관찰

③ 컨텍스트 의심 신호
   - 청크 수×길이가 모델 윈도 예산 초과로 뒤가 잘림
   - 정답 청크가 프롬프트 "중간"에 묻힘(lost-in-middle) → 답이 앞/뒤 청크만 반영
   진단: 컨텍스트에 실제 들어간 텍스트를 로깅해 gold 포함·위치 확인

④ 생성 의심 신호
   - 컨텍스트에 명백히 답이 있는데 모델이 무시/창작
   - 인용을 달면 엉뚱한 청크를 가리킴(→ [[Citation-Attribution]] 인용 환각)
   진단: context_recall은 높은데 faithfulness가 낮음(§5)
```

## 5. context_recall vs faithfulness — 점수로 단계 가르기

```
RAGAS 지표를 "단계 귀속용"으로 읽는다([[Evaluation]]):
  context_recall  낮음 → 검색층(①②). 정답 근거가 컨텍스트에 안 들어옴
  context_recall  높음 + faithfulness 낮음 → 생성층(④). 근거는 왔는데 답이 근거를 벗어남
  context_precision 낮음 → 노이즈 과다(③에 가까움). 관련 없는 청크가 답을 흐림
한 답이 아니라 "실패 답 묶음"을 이 두 축으로 분류하면 어느 단계가 지배적인지 보인다.
```

> [!warning] 집계 점수는 방향만, 단정은 단일 케이스로
> RAGAS는 "어느 단계가 자주 깨지나"의 방향을 주지만, judge 기반이라 노이즈가 있다([[Evaluation]] §9 judge 편향). 특정 오답의 최종 귀속은 §3 gold 주입으로 직접 확인한다.

## 6. 흔한 오진 (안티패턴)

```
✗ 검색 문제를 프롬프트로 덮으려 함 — 정답이 컨텍스트에 없으면 어떤 프롬프트도 못 살림
✗ "환각이다"라고 단정 — ④처럼 보여도 실은 ① recall 실패가 대부분(근거가 안 왔다)
✗ top-k만 키움 — 노이즈·비용↑, lost-in-middle 악화(③). 리랭킹이 정답일 때가 많음
✗ 청크 크기만 만짐 — 측정 없이 바꾸면 다른 질문이 깨짐 → [[Evaluation]]로 회귀 확인
✗ 단일 케이스로 파이프라인 전체 결론 — 케이스를 모아 단계별 빈도로 우선순위
```

## 7. 트레이드오프·원칙

```
디버깅 가능성 자체가 설계 요건:
  - 검색 후보·점수·최종 컨텍스트를 로깅해야 사후 귀속이 가능(관측 가능성)
  - 인용/provenance를 붙이면 ④ 판별이 쉬워짐([[Citation-Attribution]])
비용: gold 주입·단계 로깅은 수작업/저장 비용 → 회귀 셋(자주 깨지는 질문)에 집중
원칙: "틀렸다"에서 멈추지 말고 4단계 중 하나로 귀속될 때까지 좁힌다.
      고치는 곳이 정해져야 고쳐진다.
```

## 8. RAG 노트군에서의 위치

```
품질 레버(어떻게 잘 만드나): 청킹·리랭킹·쿼리변환·인입
측정(얼마나 잘 되나):        [[Evaluation]] (집계·드리프트·A/B)
진단(이건 왜 틀렸나):        이 노트 — 측정과 레버 사이를 잇는 "책임 귀속"
신뢰(답을 믿을 근거):        [[Citation-Attribution]] (grounding 검증)
```

- [[Citation-Attribution]] — grounding 검증: 생성층(④) 판별·인용 환각
- [[Reranking-Retrieval]] — 순위(②) 실패의 수정 위치
- [[Chunking-Strategies]] — 검색 누락(①)의 상류 원인·수정 위치
- [[Query-Transformation]] — 어휘 불일치 recall(①) 보강(HyDE·multi-query)
- [[RAG-Ingestion-Pipeline]] — 파싱 유실로 인한 recall(①) 상류 원인
