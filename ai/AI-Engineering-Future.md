# AI 시대의 소프트웨어 엔지니어링 — 도메인 지식과 설계로의 수렴

> [!note] 이 노트는 개인적인 관점과 생각을 정리한 것이다.

---

## 핵심 명제

> **코딩 실력은 AI로 인해 상향 평준화된다. 차별화 요소는 도메인 지식과 설계 역량이다.**

---

## 1. AI가 바꾸는 것 — 코딩의 민주화

```
과거:
  좋은 엔지니어 = 알고리즘 · 자료구조 · 언어 숙련도
  → 구현 자체가 장벽 (코딩을 할 줄 아냐 / 모르냐)

현재(2026):
  코드 생성 → Claude, GPT, Copilot이 즉시 제공
  버그 수정 → AI가 패턴 인식으로 대부분 해결
  보일러플레이트 → 생성형 AI가 초 단위 생성
  
  결과: "코드를 잘 짜는 능력"의 희소성 하락
        → 구현 장벽이 낮아지면서 더 많은 사람이 개발 가능
```

---

## 2. AI가 바꾸지 못하는 것 — 도메인 지식

```
AI는 범용 지식을 갖지만, 특정 도메인의 맥락을 모른다.

예시:
  물류 도메인:
    "주문이 하나 취소됐을 때 창고 재고, 배송차량 루트,
     세금계산서, 반품 처리, 고객 포인트가 동시에 어떻게 바뀌는가"
    → AI는 코드를 짜지만, 이 연결 관계를 모른다

  의료 도메인:
    "처방전 하나가 EMR, 건강보험공단 청구, 약국 재고,
     환자 이력에 어떻게 파급되는가"
    → 도메인 전문가만이 요구사항을 정의할 수 있다

  CCTV/물리보안:
    "ONVIF Profile T의 메타데이터 스트림이
     PTZ 제어 타임아웃과 어떻게 연동되어야 하는가"
    → 현장 경험이 있어야 올바른 설계 가능
```

---

## 3. 귀결점 — 설계와 기획으로의 전환

```
AI 시대에 가치 있는 엔지니어 역량:

1. 도메인 깊이 이해
   → "이 비즈니스에서 어떤 데이터가 어떻게 흘러야 하는가"
   → DDD의 Bounded Context, Aggregate 설계가 더 중요해진다

2. 요구사항 추출 · 명세화
   → 사용자/이해관계자의 암묵적 니즈를 구조화
   → "이걸 구현해"가 아닌 "이 문제를 어떻게 풀어야 하는가"

3. 상품화 기획
   → 도메인 지식 + 기술 이해로 실현 가능한 제품 설계
   → "AI에게 무엇을 만들라고 지시할 것인가"

4. 아키텍처 판단
   → 트레이드오프를 이해하는 설계 결정
   → "지금 이 선택이 6개월 후 어떤 결과를 낳는가"

[역할 변화]
  과거: 코드 작성자 (Implementer)
  미래: 문제 정의자 + 시스템 설계자 (Problem Definer + Architect)
```

---

## 4. DDD가 더 중요해지는 이유

```
Domain-Driven Design의 핵심:
  코드 구조를 비즈니스 도메인 언어와 일치시키는 것

AI 시대에:
  AI가 Ubiquitous Language를 이해하려면 → 명확한 도메인 모델이 필요
  AI에게 컨텍스트를 주려면 → Bounded Context 경계가 명확해야 함
  AI가 생성한 코드를 검토하려면 → 도메인 지식으로 의도 파악 가능

결국: AI를 잘 쓰는 사람 = 도메인을 잘 아는 사람
```

---

## 5. 실천 방향

```
[지금 쌓아야 할 것]

도메인 지식:
  → 특정 산업 (물류, 의료, 금융, 보안) 깊이 파고들기
  → 현업 프로세스와 데이터 흐름 이해
  → 도메인 전문가와 대화할 수 있는 언어 습득

설계 역량:
  → DDD: Bounded Context, Aggregate, Domain Event
  → 시스템 아키텍처 패턴 (CQRS, Event Sourcing, Saga)
  → "왜 이렇게 설계했는가"를 설명할 수 있는 근거

기획력:
  → 도메인 문제를 제품으로 전환하는 사고
  → 기술 실현 가능성 + 비즈니스 가치의 교차점 찾기

[불필요해지는 것]
  → 특정 언어 문법 암기
  → 보일러플레이트 코드 작성 능력
  → 구글링으로 찾을 수 있는 지식
```
