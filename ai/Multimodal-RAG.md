# 멀티모달 RAG — 시각 문서 검색 (ColPali류)

> [!summary] 한 줄 요약
> 실무 문서는 **표·차트·레이아웃이 있는 PDF**다 — 텍스트만 뽑아 검색하면 표·도표에서 정보가 깨진다. 해법은 ① 레이아웃 인식 **파싱→텍스트화** 후 일반 RAG, ② **OCR 없이 페이지를 이미지로 임베딩**해 검색(ColPali류). 생성 단계도 검색된 이미지를 VLM에 직접 넣어 답한다.

> 모델 측(ViT·이미지 토큰)은 [[VLM]], 텍스트 검색 파이프라인은 [[RAG]]·[[Chunking-Strategies]]·[[Reranking-Retrieval]]. 이 노트는 **검색 측 멀티모달**.

---

## 1. 왜 — 텍스트-only RAG가 실문서에서 실패

```
실무 코퍼스: PDF·스캔·표·차트·다단 레이아웃·수식
텍스트 추출(OCR/파서)의 손실:
  - 표 → 행/열 관계 깨짐(셀이 줄글로 뭉개짐)
  - 차트/그림 → 텍스트로 못 옮김(아예 사라짐)
  - 레이아웃(2단·박스·캡션) → 읽기 순서 꼬임
→ 검색 단위(청크)가 이미 손상 → 그 위 RAG가 다 틀림
```

## 2. 두 가지 접근

### 2.1 파싱 → 텍스트화 후 일반 RAG
```
레이아웃 인식 파서로 구조 보존:
  표 → Markdown/HTML 표로 직렬화(행·열 유지)
  그림/차트 → VLM으로 캡션·요약 생성해 텍스트로 보강
  레이아웃 → 읽기 순서·섹션 복원(document-structure 청킹, [[Chunking-Strategies]])
장점: 기존 텍스트 RAG 스택 재사용 / 단점: 파싱 품질에 종속·차트 정보 일부 손실
```

### 2.2 OCR-free 시각 검색 (ColPali류) ⭐
```
페이지를 "이미지 그대로" 임베딩해 검색 — 파싱·OCR 없이
ColPali / ColQwen (VLM 기반 late-interaction):
  - 페이지를 VLM에 넣어 패치(이미지 조각)별 임베딩 생성
  - 쿼리는 토큰별 임베딩
  - late-interaction(MaxSim): 쿼리 토큰 ↔ 페이지 패치 최대유사도 합산으로 점수
  → 표·차트·레이아웃을 "본 채로" 검색(텍스트 추출 손실 회피)
이름 유래: ColBERT의 late-interaction을 PaliGemma(VLM)에 적용 = ColPali
```

## 3. 생성 단계 — 검색된 이미지를 VLM에 직접

```
검색이 "관련 페이지 이미지"를 반환 → 그 이미지를 VLM에 직접 입력해 답 생성
  → 표·차트를 모델이 직접 "보고" 답(텍스트 변환 손실 없음)
즉 검색·생성 둘 다 멀티모달(VLM) — [[VLM]] 추론 비용↑
```

## 4. 트레이드오프 ⚠️

```
시각 검색(ColPali류):
  + 파싱·OCR 손실 회피, 표·차트·레이아웃 강함
  - 임베딩이 패치 다수(late-interaction) → 저장량·인덱스 크기↑(텍스트 단일 벡터보다 큼)
  - 검색·생성 모두 VLM → 지연·비용↑(GPU)
파싱→텍스트:
  + 저렴·기존 스택·빠름
  - 파서 품질 한계·차트 손실
결정: 표·차트가 핵심이고 정확도 중요 → 시각 검색 / 텍스트 위주 문서 → 파싱 RAG(+표만 보강)
하이브리드: 텍스트 RAG 기본 + 표·도표 페이지만 시각 검색으로 보강
```

## 5. RAG 노트군에서의 위치

```
검색 단위 축: 텍스트 청크 → (이 노트) 페이지 이미지/패치
품질 레버(쿼리변환·청킹·리랭킹)는 텍스트·시각 공통으로 적용 가능
  (예: 시각 검색 결과도 리랭킹 가능, late-interaction 자체가 정밀 점수)
모델 측 VLM([[VLM]])이 검색·생성 양쪽의 엔진
```
