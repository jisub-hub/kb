# 에이전트 보안 & 샌드박싱

> [!summary] 한 줄 요약
> 자율 에이전트는 **셸·파일·웹에 실제 액션**을 한다 — 챗봇과 달리 "잘못된 출력"이 아니라 "잘못된 실행"이 사고다. 핵심은 ① **최소 권한·격리**(샌드박스), ② **승인 게이트**(위험 액션은 사람 확인), ③ **프롬프트/메모리 인젝션 → 도구 남용 방어**, ④ **감사 로그**. 자율성과 위험은 비례한다.

---

## 1. 왜 에이전트 보안은 다른가

```
챗봇:   잘못 → 잘못된 "텍스트" (사람이 거름)
에이전트: 잘못 → 잘못된 "실행" (파일 삭제·송금·외부 요청·코드 실행)
        → 부수효과가 비가역적일 수 있음

자율성↑ = 권한↑ = 위험↑
  도구 호출 < 셸 실행 < 자가개선(스스로 코드 작성·실행)
```

[[Autonomous-Agent-Runtimes]](OpenClaw 등)·[[Hermes-Agent]]의 "권한·샌드박싱 필수" 경고를 실체화한 문서.

---

## 2. 위협 모델

| 위협 | 설명 |
|------|------|
| **프롬프트 인젝션** | 사용자/웹 콘텐츠가 "지시"로 둔갑 → 의도치 않은 도구 호출 |
| **간접 인젝션** | 에이전트가 읽은 문서·웹페이지에 숨은 명령 ("이전 지시 무시…") |
| **Memory Poisoning** | 악성 입력이 KB에 저장 → 미래 호출 지속 오염 → [[Multi-Tenant-Memory-Agent]] |
| **도구 남용** | 정상 도구를 악의적 인자로 (rm -rf, 임의 URL 요청 SSRF) |
| **권한 상승** | 한 도구로 자격증명·키 탈취 → 더 넓은 접근 |
| **데이터 유출** | 민감정보를 외부 요청·로그로 빼냄 |
| **자가개선 폭주** | 스스로 작성한 코드가 통제 벗어남 |

---

## 3. 방어 — 최소 권한 & 격리 (1순위)

```
최소 권한 (least privilege):
  - 에이전트에 "필요한 도구만" 노출 (전체 셸 대신 특정 명령 화이트리스트)
  - 도구별 스코프 제한 (읽기 전용 vs 쓰기, 특정 디렉토리만)
  - 자격증명은 단기·범위 한정 토큰 (장기 키 직접 노출 금지)

격리 (sandboxing):
  - 컨테이너/VM/마이크로VM(gVisor·Firecracker)에서 실행
  - macOS sandbox-exec(seatbelt), Linux seccomp·namespace
  - 네트워크 egress 화이트리스트 (임의 외부 요청 차단 → SSRF 방어)
  - 파일시스템 격리 (작업 디렉토리만 마운트, 읽기전용 루트)
  - 리소스 제한 (CPU·메모리·실행시간 — 폭주 방지)
```

> 원칙: **에이전트는 격리된 환경에서, 필요 최소 권한으로만 실행.** "전체 셸 + 풀 권한"은 사고의 지름길.

---

## 4. 방어 — 승인 게이트 (Human-in-the-Loop)

```
액션 위험도 분류:
  자동 허용:   읽기·조회·검색 (부수효과 없음)
  승인 필요:   쓰기·삭제·송금·외부 전송·코드 실행 (비가역·민감)
  항상 차단:   자격증명 접근·시스템 변경·대량 삭제

위험 액션 = 실행 전 사람 확인 (요약 제시 → 승인/거부)
배치·대량 작업 = 드라이런(dry-run) 먼저
```

```java
// 위험 도구 호출 가로채기
if (tool.riskLevel() >= APPROVAL_REQUIRED) {
    var decision = approvalGate.request(tool, args, summary);  // 사람에게
    if (!decision.approved()) return ToolResult.denied();
}
auditLog.record(tenant, tool, args, decision);   // 감사 (6절)
```

## 5. 방어 — 인젝션 & 도구 남용

```
프롬프트/간접 인젝션:
  - 신뢰 경계 분리: 시스템 지시 vs 사용자 입력 vs 외부 콘텐츠 명확히 구분
    (외부 문서·웹은 "데이터"로만, "지시"로 해석 금지)
  - 구조화 입력(XML/델리미터)로 콘텐츠 격리 → [[Prompt-Engineering]]
  - 출력의 도구 호출을 스키마·정책으로 검증 (허용 도구·인자 범위)

도구 남용:
  - 인자 검증·새니타이즈 (경로 traversal·셸 메타문자·URL 화이트리스트)
  - 도구별 rate limit (폭주·반복 호출 차단) → [[Rate-Limiting]]
  - 루프 한도·토큰 예산 (무한 에이전트 루프 방지) → [[Agent-Harness]]

Memory Poisoning:
  - KB 저장 전 검증, 사용자 발화/시스템 사실 신뢰도 분리
  - 저장된 기억을 "지시"가 아닌 "참고"로 주입 → [[Multi-Tenant-Memory-Agent]]
```

---

## 6. 방어 — 관찰성 & 감사

```
감사 로그 (모든 도구 호출·승인·결과):
  who(tenant/user) · what(tool+args) · when · 승인여부 · 결과 · trace_id
  → 사후 추적·이상 탐지·규제 대응 (→ [[AOP-Logging]] MDC traceId)

모니터링:
  - 비정상 도구 호출 빈도·패턴 알림
  - 권한 거부·차단 이벤트 추적
  - 비용 폭증(토큰·API) 알림 → [[LLMOps]]
```

## 7. 자가개선(self-improving) 통제

자율 런타임(OpenClaw·Hermes Agent)은 **스스로 코드를 작성·실행**해 능력을 확장한다. 가장 강력하면서 가장 위험.

```
- 생성 코드도 동일 샌드박스·권한 정책 적용 (예외 없음)
- 새 스킬은 격리 환경에서 검증 후 승인 → 프로덕션 반영
- 생성 코드 리뷰 게이트 (자동 정적분석 + 위험 시 사람)
- 능력 확장 범위 제한 (허용 카테고리 밖 도구 생성 금지)
```

---

## 8. 체크리스트

```
□ 최소 권한 — 필요한 도구만, 스코프 제한
□ 격리 실행 — 컨테이너/VM, 네트워크·FS·리소스 제한
□ 승인 게이트 — 위험 액션은 사람 확인, 대량은 드라이런
□ 신뢰 경계 — 외부 콘텐츠는 데이터로만 (지시 해석 금지)
□ 인자 검증 — 경로·셸·URL 새니타이즈, SSRF 방어
□ rate limit · 루프 한도 · 토큰 예산
□ Memory poisoning 방어 — 저장 검증, 기억은 참고로만
□ 감사 로그 — 모든 도구 호출·승인 기록 (trace_id)
□ 자가개선 코드도 동일 정책 + 리뷰 게이트
□ 자격증명 — 단기·범위한정 토큰, 직접 노출 금지
```

---

## 9. 정리

```
에이전트 보안 = "잘못된 출력"이 아니라 "잘못된 실행"을 막는 것.
1순위: 최소 권한 + 격리(샌드박스)
2순위: 위험 액션 승인 게이트(human-in-the-loop)
3순위: 인젝션→도구남용 방어(신뢰 경계·인자 검증·rate limit)
상시:  감사 로그 + 모니터링
특수:  자가개선 코드도 예외 없이 동일 통제
원칙:  자율성↑ = 권한↑ = 위험↑ → 권한을 최소로, 위험은 게이트로.
```
