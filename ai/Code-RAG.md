# 코드베이스 RAG — AST·심볼 인지 검색

> [!summary] 한 줄 요약
> 코드는 산문과 검색 특성이 **근본적으로 다르다**. 고정 크기 청킹은 함수·클래스를 중간에서 자르고, 벡터 유사도만으로는 "이 함수가 어디서 호출되나"를 못 찾는다. 코드 RAG는 ① **AST 기반 청킹**(함수/클래스 경계 + import·시그니처 동반) ② **심볼 인덱스·콜그래프**(정의↔사용 추적, LSP/ctags) ③ **코드 전용 임베딩**을 결합한다. 핵심은 *질의 의도*(정의 찾기 vs 사용처 찾기 vs 설명)마다 검색 경로가 다르다는 것. 함정은 **동명 심볼·오버로드·생성 코드·테스트 노이즈**.

> 산문 청킹은 [[Chunking-Strategies]], 일반 임베딩은 [[Embedding]], 엔티티 그래프는 [[GraphRAG]], 코드 에이전트 실행은 [[Orchestrating-Claude-Code]]. 이 노트는 **코드 구조를 인지하는 검색**.

---

## 1. 왜 산문 RAG가 코드에 실패하나

```
고정/재귀 청킹: 토큰 N개로 자르면 함수 본문 중간이 잘림
  → 청크가 시그니처 없이 본문만 / 클래스 맥락 없는 메서드 조각 → 검색·이해 실패
벡터 유사도의 한계:
  "이 함수 어디서 쓰여?" → 의미 유사가 아니라 "정확한 심볼 참조" 문제
  "BigDecimal 반올림 버그" → 코드 토큰은 자연어 질의와 어휘가 어긋남
근본: 코드엔 산문에 없는 정밀 구조(AST·심볼·import·콜그래프)가 있고
      검색이 이를 무시하면 정확도가 무너진다
```

## 2. AST 기반 청킹 ⭐

```
파서/tree-sitter로 구문 트리를 떠서 "의미 단위"로 자른다:
  - 함수/메서드·클래스·인터페이스 = 자연스러운 청크 경계
  - 각 청크에 맥락 동반: 파일 경로·언어·소속 클래스·import·시그니처
    (Contextual Retrieval의 코드판 — 청크가 자기 위치를 알게)
  - 너무 큰 함수는 논리 블록으로 분할, 너무 작은 건 인접 병합
효과: 청크 = 독립적으로 이해·실행 가능한 코드 단위 → 검색·생성 정확↑
도구: tree-sitter(다언어 파서), 언어별 LSP
```

## 3. 검색 의도별 경로 — 벡터만으론 부족 ⭐

| 질의 의도 | 예 | 올바른 검색 |
|-----------|-----|-------------|
| 정의 찾기 | "`OrderService` 어디 정의?" | 심볼 인덱스 lookup(정확) |
| 사용처 찾기 | "`refund()` 누가 호출?" | 역참조/콜그래프(LSP find-references) |
| 의미·설명 | "환불 한도 검증 로직" | 벡터·하이브리드 검색 |
| 영향 범위 | "이 DTO 바꾸면?" | 의존성 그래프 traversal |

```
원칙: 코드 RAG는 "벡터 검색 + 심볼/그래프 검색"의 하이브리드.
  심볼 정확 질의를 임베딩에 맡기면 동명·근사로 빗나감 → 결정적 인덱스가 1차
  repo가 크면 전체 임베딩은 비현실적·갱신 비쌈 → 심볼 인덱스가 싸고 정확한 1차 필터
```

## 4. 코드 임베딩·의존성 확장

```
임베딩: 자연어-코드 정렬 모델(코드·docstring 쌍 학습) — 일반 텍스트 모델보다
        "자연어 질의 ↔ 코드" 매칭에 강함. 단 도메인/언어 편향 존재 → 측정([[Evaluation]])
의존성 확장(grounding 보강):
  검색된 청크가 부르는 함수·타입 정의를 import/콜그래프로 따라가 함께 컨텍스트에 동반
  → "이 함수만" 보여주면 LLM이 호출 대상 시그니처를 몰라 환각
  → 1-hop 정의 동반이 코드 생성/설명 정확도를 크게 올림
```

## 5. ★ 함정 ★

```
✗ 동명 심볼·오버로드: 같은 이름 함수가 여러 개 → 심볼 검색이 엉뚱한 정의 반환
   완화: 네임스페이스·시그니처·파일 경로로 구분(정규화된 심볼 ID)
✗ 생성·벤더 코드: 빌드 산출물·node_modules·generated가 인덱스를 오염
   완화: .gitignore/경로 규칙으로 색인 제외
✗ 테스트 노이즈: 테스트가 프로덕션 심볼을 다량 참조 → 사용처 검색이 테스트로 도배
   완화: src vs test 가중·필터
✗ 버전 드리프트: 코드가 자주 바뀜 → 증분 재색인 필수(파일 해시 변경분만) → [[RAG-Ingestion-Pipeline]]
✗ 청크가 맥락 상실: 시그니처·import 없는 본문 청크는 검색·이해 모두 실패(§2로 방지)
```

## 6. RAG 검색 패러다임 지도에서의 위치

```
산문 문서 검색:   [[Chunking-Strategies]]·[[Reranking-Retrieval]] (의미 청크)
정형 데이터 검색: [[Structured-Data-RAG]] (text-to-SQL)
관계 그래프 검색: [[GraphRAG]] (엔티티/커뮤니티)
코드 구조 검색:   이 노트 — AST·심볼·콜그래프(결정적 인덱스 + 벡터 하이브리드)
공통: 데이터 형태가 "검색 단위와 인덱스 종류"를 정한다 → RAG ≠ 단일 벡터 검색
```
