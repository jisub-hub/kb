# 컨텍스트 조립·압축 — 리랭킹과 생성 사이의 레버

> [!summary] 한 줄 요약
> 리랭킹이 청크를 *관련도 순위*까지 매겨도, 그것을 **프롬프트에 어떻게 배치·압축하느냐**는 별개 단계다. 같은 top-k라도 ① **배치 순서**(lost-in-middle 대응: 중요한 청크를 head·tail에) ② **토큰 예산 배분**(몇 개·parent 확장 vs 윈도우) ③ **프롬프트 압축**(LLMLingua류로 의미 보존하며 토큰↓) ④ **중복 병합**(overlap·near-dup 제거)에 따라 답 품질이 달라진다. 검색의 마지막 단계이자, 가장 자주 방치되는 레버.

> 관련도 정렬은 [[Reranking-Retrieval]], 분할은 [[Chunking-Strategies]], 단계 실패 진단은 [[RAG-Failure-Debugging]], KV 캐시 재사용은 [[CAG-Cache-Augmented-Generation]]. 이 노트는 **선택된 청크의 프롬프트 레이아웃·압축**.

---

## 1. 왜 별개 단계인가 — 관련도 순서 ≠ 배치 순서

```
리랭킹 출력: [c1, c2, c3, c4, c5]  (관련도 내림차순)
순진한 조립: 그 순서로 프롬프트에 이어 붙임
문제: LLM은 컨텍스트 "가운데"의 정보를 잘 놓친다(lost-in-the-middle)
  → 가장 약한 c5가 끝에, c3·c4가 가운데 묻혀 무시될 수 있음
즉 "무엇을 뽑을까"(검색)와 "어떻게 놓을까"(조립)는 다른 문제다
```

## 2. 배치 순서 — lost-in-middle 대응 ⭐

```
관찰: 모델은 컨텍스트의 처음과 끝을 가운데보다 잘 활용
전략: 가장 중요한 청크를 head와 tail에, 덜 중요한 건 가운데로
  예) 관련도 [1,2,3,4,5] → 배치 [1, 3, 5, 4, 2]  (1·2를 양 끝)
또한:
  - 질문을 컨텍스트 뒤(생성 직전)에 다시 두면 최신 주의 확보
  - 청크마다 출처·순번 메타를 붙여 인용·추적 가능([[Citation-Attribution]])
원칙: 리랭킹 점수는 "선택"에 쓰고, 배치는 모델의 위치 편향을 고려해 따로 정한다
```

## 3. 토큰 예산 배분

```
컨텍스트 윈도우는 유한 → 청크 수·길이·다른 요소(시스템·이력·질문)와 경쟁
- 청크 개수 k: 많을수록 recall↑ 그러나 노이즈·lost-in-middle·비용↑
- parent-document 확장: 작게 검색→큰 부모 전달 시 예산 급증([[Chunking-Strategies]])
- 멀티턴이면 이력과 예산 경쟁([[Conversational-RAG]])
판단: "정답에 필요한 최소 컨텍스트"가 목표 — 더 넣는다고 더 정확하지 않다
      (노이즈가 늘면 오히려 faithfulness 저하 → [[RAG-Failure-Debugging]] ③)
```

## 4. 프롬프트 압축 (LLMLingua류)

```
아이디어: 청크 텍스트를 "의미 보존하며" 토큰 단위로 압축해 더 많이/싸게 담기
  - 정보량 낮은 토큰(불용어·중복 구절)을 작은 모델로 판단해 제거
  - 질문 연관 낮은 문장 드롭(query-aware 압축)
효과: 같은 윈도우에 더 많은 근거 / 같은 근거를 더 싸게
함정:
  - 과압축 → 핵심 사실·수치·부정어("not") 유실 → 오답
  - 압축 자체가 추가 지연·비용(작은 모델 호출) → 이득과 상쇄 확인
  - 코드·표·인용 대상은 압축 금물(구조·정확성 깨짐)
```

## 5. 중복 병합·정리

```
청킹 overlap(겹침)·유사 문서로 near-duplicate 청크가 같이 뽑히기 쉽다:
  - near-dup 제거(임베딩 유사도·해시)로 노이즈·예산 낭비↓
  - 인접 청크 병합(auto-merging)으로 끊긴 맥락 복원
  - 형식 정리: 청크 사이 명확한 구분자, 출처 헤더(파일·섹션)
효과: 같은 예산에 "서로 다른" 근거를 더 담아 답의 커버리지↑
```

## 6. RAG 파이프라인에서의 위치

```
검색 → 리랭킹(관련도 순위) → [이 단계: 조립·압축] → 생성
  선택은 끝났고, "프롬프트라는 한정된 무대에 어떻게 올리나"가 여기서 결정
무대 연출 3레버: 순서(lost-in-middle)·예산(얼마나)·압축(밀도)
원칙: 검색을 아무리 잘해도 조립이 나쁘면 생성이 못 받는다(마지막 1단계)
```
