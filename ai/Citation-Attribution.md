# 인용·출처 귀속 (Citation & Attribution)

> [!summary] 한 줄 요약
> RAG가 **답이 어느 근거에서 왔는지 표시·검증**하게 만드는 grounding 레이어. 검색·생성이 맞아도 "출처 추적"이 없으면 사용자가 신뢰·검증할 수 없다. Faithfulness *측정*([[Evaluation]])과 달리, 이건 **생성 시점에 근거를 부착하고 사후 검증**하는 운영 패턴. Self-RAG의 IsSupported([[Agentic-RAG]])를 파이프라인 밖으로 꺼낸 것.

> 검색 파이프라인은 [[RAG]], 근거 충실성 *측정*은 [[Evaluation]], 능동 자기검증은 [[Agentic-RAG]]. 이 노트는 **"답↔근거 연결을 표시·검증"**.

---

## 1. 왜 — 검색·생성이 맞아도 신뢰가 안 된다

```
RAG 답변: "환불은 30일 이내 가능합니다."   ← 이게 어느 문서/청크에서 왔나?
근거 추적이 없으면:
  - 사용자가 검증 불가(맞말처럼 보이는 환각을 못 거름)
  - 규제/감사 대응 불가(금융·의료·법무는 출처 의무)
  - 디버깅 불가(틀린 답이 검색 탓인지 생성 탓인지 모름)
해법: 답의 각 주장에 출처를 부착(citation) + 그 출처가 실제로 뒷받침하는지 검증(attribution)
```

> 용어: **citation**(답에 출처 표시) ⊂ **attribution**(주장↔근거의 실제 뒷받침 관계). 표시는 쉬워도 *올바른* 표시가 어렵다.

## 2. 인라인 인용 생성

```
검색된 청크에 ID 부여 → 프롬프트에 "[1] {chunk1}\n[2] {chunk2} ..." 로 주입
→ "각 문장 끝에 사용한 출처를 [n]으로 표기하라" 지시
출력: "환불은 30일 이내 가능하다 [2]. 단 디지털 상품은 제외된다 [5]."
구조화 출력(권장): {answer, citations:[{claim_span, source_ids:[2]}]} JSON 스키마 강제
  → 자유 텍스트 [n]보다 파싱·검증·렌더링이 안정적
```

- 청크 ID는 검색 단계에서 부여([[Chunking-Strategies]] 청크 = 인용 단위). 청크가 너무 크면 인용이 뭉뚱그려짐 → **인용 단위 = 검증 가능 최소 단위**가 이상적.

## 3. 스팬 수준 귀속 (span-level attribution)

```
문서 단위(D)   → 청크 단위([n]) → 문장 단위 → 스팬(정확한 인용 구절)
오른쪽으로 갈수록 검증·신뢰 정밀↑, 비용·난이도↑
스팬 귀속: 답 문장 ↔ 근거의 "정확한 문장/구절"을 매핑
  방법: ① 생성모델이 직접 인용구 추출  ② 사후 별도 attribution 모델/NLI로 정렬
효용: 사용자가 클릭→원문 하이라이트로 즉시 검증(UX), 환각 인용 적발 용이
```

## 4. ★인용 환각★ — 핵심 실패 모드 ⚠️

```
실패: 모델이 "그럴듯한" 출처 번호를 붙이지만 실제론 그 청크가 주장을 뒷받침 안 함
  - [3]을 달았는데 [3]엔 그 내용이 없음(틀린 귀속)
  - 존재하지 않는 [9]를 만듦(범위 밖 인용)
  - 파라메트릭 지식으로 답하고 출처를 끼워맞춤(검색과 무관한 답에 인용 위장)
왜 위험: 인용이 있으면 사용자가 "검증됐다"고 더 믿어버림 → 환각보다 더 해로울 수 있음
```

## 5. 사후 검증 (post-hoc verification)

```
생성 후 각 (주장, 인용 출처) 쌍을 검증해 거름:
  ① 범위 검사   : 인용 ID가 실제 검색결과에 존재하는가(범위 밖 → 제거/플래그)
  ② 함의 검사   : 근거가 주장을 entail 하는가
                  - 문자열/n-gram 중첩(싸고 거침)
                  - NLI 모델(entail/neutral/contradict)
                  - LLM-as-judge "이 근거가 이 주장을 뒷받침하나?"([[Evaluation]] §9 편향 주의)
  ③ 미검증 처리 : 뒷받침 안 되는 주장 → 인용 제거·문장 삭제·"근거 없음" 표시·재생성
→ Self-RAG IsSupported를 파이프라인 밖 독립 단계로 외부화([[Agentic-RAG]] §2)
```

> [!note] 측정 vs 검증
> - [[Evaluation]] Faithfulness: **오프라인 골든셋**에서 평균 충실성 *점수* — "이 시스템 좋은가".
> - 사후 검증(이 노트): **온라인 모든 응답**에서 *개별* 인용을 게이트 — "이 답을 내보낼까".

## 6. 트레이드오프

```
이득 : 검증가능성·신뢰·감사대응·디버깅(검색탓↔생성탓 분리)
비용 : 인용 강제 시 출력 토큰·지연↑, 사후 검증은 추가 모델 호출(NLI/judge)
       → [[RAG-Latency-Reality]] 생성+검증 누적
품질 : 인용 강제가 답을 부자연스럽게/방어적으로 만들 수 있음(과인용)
원칙 : 고위험·규제 도메인은 스팬 귀속+사후 검증, 일반 QA는 청크 [n] 인용으로 충분.
       인용 단위를 검증 가능 최소 단위로(청킹과 직결).
```

## 7. RAG 신뢰 스택에서의 위치

```
검색 품질   : 청킹·리랭킹·쿼리변환 (올바른 근거를 뽑기)
생성 충실성 : 프롬프트·근거 주입 (근거 안에서 답하기 — [[RAG]])
귀속(이 노트): 답↔근거 연결 표시·검증 (어느 근거인지 + 정말 뒷받침하나)
능동 자기검증: Self-RAG IsSupported로 루프 내 재생성([[Agentic-RAG]])
측정         : Faithfulness/Recall 골든셋 평가([[Evaluation]])
```
