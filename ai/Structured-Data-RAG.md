# 구조화 데이터 RAG — text-to-SQL·툴 검색 vs 문서 검색

> [!summary] 한 줄 요약
> 문서(벡터) RAG는 *서술형 지식*엔 강하지만 **수치 집계·정확 필터·정렬·"몇 건"·최신 트랜잭션**엔 구조적으로 실패한다. 이런 질의는 구조화 데이터에 **text-to-SQL/툴 호출**로 답해야 한다. 핵심은 ① **라우팅**(서술형→문서 vs 분석형→SQL) ② **스키마 링킹**(NL→테이블·컬럼, 관련 스키마만 주입) ③ **생성 SQL 검증·실행 가드**(read-only·LIMIT·EXPLAIN) ④ ★**환각 SQL·잘못된 조인/집계**★. 벡터 검색과 별개 패러다임이며, 실무 KB는 둘을 라우팅으로 합친다.

> 문서 검색 파이프라인은 [[RAG]], 저장 스키마는 [[KB-Database-Schema]], 툴 호출 일반은 [[Agent-Harness]]. 이 노트는 **구조화 데이터를 자연어로 질의하는 검색측**.

---

## 1. 왜 벡터 검색은 집계 질문에 실패하는가

```
"ACME의 2023 Q3 매출 합계는?" / "지난주 가입자 중 미결제 몇 명?"
벡터 검색의 동작: 의미가 유사한 청크 top-k를 가져옴
  → 합계·카운트·정렬·최신값은 "유사한 텍스트"로 안 나온다
  → 답이 여러 행에 흩어져 있으면 top-k로는 전부 못 모음(완전성 보장 X)
  → 숫자는 임베딩 공간에서 의미적으로 가깝지 않음(123 vs 124)
근본: 문서 RAG는 "근사 의미 검색", 집계는 "정확·완전 연산"
  → 후자는 DB 엔진(SQL/집계)이 해야 할 일
```

## 2. 두 검색 패러다임 — 라우팅이 출발점 ⭐

| 질문 유형 | 예 | 올바른 검색 |
|-----------|-----|-------------|
| 서술·설명형 | "환불 정책 알려줘" | 문서 RAG(벡터/하이브리드) |
| 집계·분석형 | "이번 달 환불 총액" | text-to-SQL / 집계 쿼리 |
| 정확 필터·조회 | "주문 #1234 상태" | 구조화 조회(키 lookup) |
| 혼합 | "환불 정책과 내 환불 내역" | 둘 다 → 결과 병합 |

```
라우터: 질문을 분류(LLM 분류 or 의도 룰) → 적절한 검색 경로로 보냄
  분류 실패가 전체 실패로 직결 → 모호하면 둘 다 시도 후 병합이 안전
  툴 호출 프레임으로 보면 라우팅 = "어떤 도구를 부를까" → [[Agent-Harness]]
```

## 3. text-to-SQL 파이프라인

```
NL 질문 → (스키마 링킹) → SQL 생성 → 검증 → 실행 → 결과를 LLM에 주입 → 답
```

### 3.1 스키마 링킹 (가장 큰 레버)
```
문제: 스키마가 크면(수백 테이블) 전부 프롬프트에 못 넣음 → 컨텍스트 초과·정확도↓
해법: "관련 테이블/컬럼만" 선별해 주입 —
  - 테이블·컬럼 설명을 임베딩해 질문으로 검색(여기서 벡터 RAG가 메타에 쓰임!)
  - FK 그래프로 연관 테이블 확장
  - few-shot: 유사 질문→SQL 예시를 검색해 함께 제공
→ 즉 "스키마 자체에 대한 RAG"가 text-to-SQL 정확도를 지배
```

### 3.2 SQL 생성·검증·실행 가드 ⭐
```
생성된 SQL을 절대 무검증 실행하지 않는다:
  - 구문/스키마 검증: 파싱·EXPLAIN으로 실행 전 오류·풀스캔 탐지
  - 권한 가드: read-only 역할·뷰만 노출·테이블 화이트리스트
  - 자원 가드: 자동 LIMIT·쿼리 타임아웃·행수 상한(거대 결과 방지)
  - 자기수정 루프: 실행 에러를 LLM에 되먹여 SQL 재생성(한도 두기 → 폭주 방지)
보안: 사용자 입력이 SQL이 되는 경로 = injection 표면. 파라미터화·격리 필수
      (검색 콘텐츠 비신뢰 원칙과 같은 맥락 → [[RAG-Security]])
```

## 4. ★ 함정 — 환각 SQL·미묘한 오답 ★

```
문법은 맞는데 의미가 틀린 SQL이 가장 위험(에러도 안 나고 그럴듯한 숫자가 나옴):
  - 잘못된 조인 → 행 중복으로 SUM 부풀려짐(팬아웃)
  - GROUP BY 누락/오류 → 엉뚱한 집계 단위
  - NULL·중복 처리 미스 → COUNT가 틀림
  - 날짜/타임존·통화 단위 가정 오류
  - 컬럼 의미 오해(amount가 세전인지 세후인지)
→ "숫자가 나왔다 = 맞다"가 아니다. 사람·테스트로 검증 가능하게:
   생성된 SQL을 답과 함께 노출(provenance) → 사용자가 쿼리를 검수
   gold 질문-SQL 회귀셋으로 정확도 추적([[Evaluation]])
```

## 5. 하이브리드 — 문서 + 구조화 결합

```
실무 질문은 섞여 있다: "환불 정책이 뭐고, 내 지난달 환불은 얼마야?"
  → 문서 RAG(정책 텍스트) + SQL(개인 거래 집계) 결과를 함께 컨텍스트에 주입
  → 한 답으로 합성
주의: 권한 경계가 다름 — SQL 결과는 그 사용자 데이터만(ACL/RLS → [[KB-Access-Control]])
패턴: 에이전트가 도구 2개(문서검색·SQL)를 상황에 맞게 호출 → [[Agentic-RAG]]·[[Agent-Harness]]
```

## 6. RAG 노트군에서의 위치

```
문서 검색축: 청킹·임베딩·리랭킹·쿼리변환([[RAG]] 계열) — 비정형 텍스트
구조화 검색축: 이 노트 — text-to-SQL·툴 호출 — 정형 데이터
공통 도구: 스키마 링킹·few-shot 선택에 "벡터 RAG"가 재사용됨(메타 검색)
원칙: RAG ≠ 벡터 검색. RAG = "외부 지식을 검색해 생성에 주입" — 검색 방식은 데이터 형태가 정한다
```
